## 六、各任务修复方案与指引（含代码）

> 下面每一项都给出**根因 → 正确改法 → 验证方式**。代码可直接参照落地。行号以本次审查时的代码为准，改动前请先确认当前内容。

### 修复 1（P0）：`settings.py` 启动即崩溃

**根因**：`@model_validator(mode="after")` 的入参是**模型实例本身**（`self`），不是带 `.data` 的 `ValidationInfo`。现有代码 `info.data.get(...)` 在实例上取不到 `.data` → `AttributeError`。另外 `Field(default="", min_length=16)` 会让 `.env` 里空字符串 `API_SECRET_KEY=` 触发 pydantic 的 `string_too_short`，报错信息也不友好。

**正确改法**（`src/common/config/settings.py`）：去掉 `min_length`，改用干净的 after 校验，缺失时抛 `ValueError`（不要用 `sys.exit`，让 pydantic 给出结构化报错）：

```python
# 顶部导入保持
from pydantic import Field, model_validator

    # FastAPI —— 去掉 min_length，避免空值触发晦涩的 string_too_short
    api_secret_key: str = Field(
        default="",
        description="JWT signing key. 必须通过环境变量 API_SECRET_KEY 设置，≥16 位。",
    )
    api_access_token_expire_minutes: int = 120
    api_refresh_token_expire_days: int = 7
    api_host: str = "0.0.0.0"
    api_port: int = 8000

    # Sandbox
    sandbox_container_pool_size: int = 5
    sandbox_exec_timeout_sec: int = 60
    sandbox_network: str = "sandbox-net"

    # Logging
    log_level: Literal["DEBUG", "INFO", "WARNING", "ERROR"] = "INFO"

    @model_validator(mode="after")
    def _validate_api_secret_key(self) -> "Settings":
        if len(self.api_secret_key) < 16:
            raise ValueError(
                "API_SECRET_KEY 未设置或不足 16 位，请通过环境变量或 .env 配置 "
                "(生产环境建议由 KMS 注入)。"
            )
        return self
```

要点：把 `# Logging` 那行错位的注释缩进也一并修正（当前 `settings.py:76` 顶格了）；把 validator 挪到所有字段定义之后。

**验证**：
```bash
API_SECRET_KEY=dev-secret-key-0123456789 .venv/Scripts/python -c "from src.common.config.settings import get_settings; print('ok', get_settings().llm_provider)"
```
应打印 `ok openai`（不再报 AttributeError）。缺 key 时应报中文 ValueError。

### 修复 4（P1）：responder 子图接入主图

**根因**：`main_graph/graph.py` 的 `build_main_graph` 没有 responder 节点，`route_decision` 只返回 investigate/vuln_check/ignore，响应链路运行时不可达。

**正确改法**（`src/orchestration/main_graph/graph.py`）：新增 respond 节点，并让 aggregator 后按 verdict 决定是否进入响应（与修复 5 的阈值路由配合）。

```python
async def respond_node(state: MainGraphState) -> dict:
    """运行响应子图（带超时）。"""
    from src.orchestration.subgraphs.responder.graph import build_responder_subgraph
    subgraph = build_responder_subgraph()
    sub = state.get("subgraph_result") or {}
    sub_state = {
        "event_id": state["event_id"],
        "verdict": sub.get("final_verdict", "unknown"),
        "confidence": sub.get("confidence_score", 0.0),
        "iocs": state.get("raw_event", {}).get("iocs", {}),
        "playbook_draft": None,
        "operation_level": "L1",
        "approval_status": None,
        "approval_id": None,
        "execution_result": None,
    }
    try:
        result = await asyncio.wait_for(subgraph.ainvoke(sub_state), timeout=_TIMEOUT_RESPOND)
        return {"pending_action": {
            "approval_id": result.get("approval_id"),
            "execution_summary": result.get("execution_result"),
        }}
    except asyncio.TimeoutError:
        logger.warning("responder_timeout", event_id=state["event_id"])
        return {"error": "responder_timeout"}


def build_main_graph() -> StateGraph:
    graph = StateGraph(MainGraphState)
    graph.add_node("entry", entry_node)
    graph.add_node("orchestrator", orchestrator_node)
    graph.add_node("investigate", investigate_node)
    graph.add_node("vuln_check", vuln_check_node)
    graph.add_node("aggregator", aggregator_node)
    graph.add_node("respond", respond_node)          # 新增
    graph.add_node("ignore", ignore_node)

    graph.add_edge(START, "entry")
    graph.add_edge("entry", "orchestrator")
    graph.add_conditional_edges("orchestrator", route_decision, {
        "investigate": "investigate",
        "vuln_check": "vuln_check",
        "ignore": "ignore",
    })
    graph.add_edge("investigate", "aggregator")
    graph.add_edge("vuln_check", "aggregator")
    # aggregator 后按置信度分流（见修复 5）
    graph.add_conditional_edges("aggregator", route_after_verdict, {
        "respond": "respond",
        "vuln_check": "vuln_check",
        "done": END,
    })
    graph.add_edge("respond", END)
    graph.add_edge("ignore", END)
    return graph.compile()
```

**验证**：提交一条高危事件（verdict=true_positive、confidence≥0.8），确认流程进入 respond 节点并产生 approval_id。

### 修复 5（P1）：置信度阈值路由（文档 4.3.2）

**根因**：文档要求 ≥0.8→响应、0.5–0.8→漏洞验证、<0.5→归档，代码里 confidence 被记录但从不用于分流。

**正确改法**（`src/orchestration/main_graph/graph.py`，配合修复 4 的 `route_after_verdict`）：

```python
def route_after_verdict(state: MainGraphState) -> str:
    sub = state.get("subgraph_result") or {}
    verdict = sub.get("final_verdict", "unknown")
    conf = sub.get("confidence_score", 0.0) or 0.0

    # 已经是漏洞子图回来的结果，直接结束，避免环路
    if state.get("stage") == "verify":
        return "respond" if verdict == "true_positive" else "done"

    if verdict == "true_positive" and conf >= 0.8:
        return "respond"
    if 0.5 <= conf < 0.8:
        return "vuln_check"      # 可疑 → 漏洞验证
    return "done"                # <0.5 → 归档
```

注意防环路：漏洞子图 → aggregator → 若再判 vuln_check 会死循环，故用 `stage` 标记来源（在 `vuln_check_node` 返回里加 `"stage": "verify"`）。`recursion_limit` 也应设上限兜底。

**验证**：构造三档 confidence 的研判结果（mock 或真实），确认分别走 respond / vuln_check / 归档。

### 修复 6（P1）：GraphRAG 真实向量 + Milvus 索引

**根因 A**：`cti_analyst.py:44` 用 `[0.0]*768` 零向量查询，BGE 嵌入模型没接入 → 向量检索恒为空。
**根因 B**：`milvus_client.py` 的 `_ensure_collection` 建了 collection 但**没建索引**，`search` 里 `collection.load()` 在无索引时会失败。

**正确改法 A**——新增嵌入模块并在检索前生成真实向量：

```python
# src/knowledge/graphrag/vector/embedding.py（新增）
from functools import lru_cache

@lru_cache(maxsize=1)
def _get_model():
    from sentence_transformers import SentenceTransformer  # 加入依赖
    return SentenceTransformer("BAAI/bge-large-zh-v1.5")

def embed(text: str) -> list[float]:
    # bge 中文检索建议加指令前缀
    vec = _get_model().encode("为这个句子生成表示以用于检索相关文章：" + text,
                              normalize_embeddings=True)
    return vec.tolist()
```
`cti_analyst._query_graphrag` 里把 `mock_embedding = [0.0]*768` 换成 `embed(" ".join(ioc_values))`。依赖加入 `pyproject.toml`：`sentence-transformers`（首次会下载模型，离线环境需预置）。

### 修复 7（P1）：HITL 审批闭环（分级 / 超时 / 升级 / 落审计）

**根因**：`hitl_handler.py` 的 `hitl_approval_node` 对 L3+ 直接返回 `pending`，边直连 `execute_response`，`pending != approved` 导致 L3+ 永远被跳过；`_LEVEL_TIMEOUT` 定义了但没用；无 L4 双人、L5 双主管；execute_response 不写 ES 审计。

**正确改法**：把审批做成“挂起—等待—恢复”，用 LangGraph 条件边在 hitl 节点后按状态分流；审批结果由 API 端点写回内存队列（已有 `resolve_approval_by_event_id`）后，子图轮询/恢复。MVP 可用“子图内轮询 + 超时默认拒绝”：

```python
# responder/graph.py —— hitl 后按审批结果分流
def route_after_hitl(state):
    return "execute_response" if state.get("approval_status") == "approved" else "reject"

graph.add_conditional_edges("hitl_approval", route_after_hitl, {
    "execute_response": "execute_response",
    "reject": "reject_response",   # 新增：写审计 + 升级告警
})
```
```python
# hitl_handler.py —— L3+ 轮询等待 + 超时默认拒绝
import asyncio
async def hitl_approval_node(state):
    level = state.get("operation_level", "L1")
    if level in ("L1", "L2"):
        return {"approval_status": "approved", "approval_id": None}

    approval_id = str(uuid.uuid4())
    timeout_sec = _LEVEL_TIMEOUT.get(level, 300)
    n_required = {"L3": 1, "L4": 2, "L5": 2}.get(level, 1)   # 分级人数
    await _push_approval_card(state["event_id"], approval_id, state.get("playbook_draft") or {}, level, timeout_sec)
    _pending_approvals[approval_id] = {"event_id": state["event_id"], "status": "pending",
                                       "required": n_required, "approvals": []}
    # 轮询等待（真实生产应换成 interrupt/resume 或事件驱动）
    waited, step = 0, 5
    while waited < timeout_sec:
        p = _pending_approvals[approval_id]
        if p["status"] == "approved" and len(p["approvals"]) >= p["required"]:
            return {"approval_id": approval_id, "approval_status": "approved"}
        if p["status"] == "rejected":
            return {"approval_id": approval_id, "approval_status": "rejected"}
        await asyncio.sleep(step); waited += step
    _pending_approvals[approval_id]["status"] = "timeout"
    return {"approval_id": approval_id, "approval_status": "timeout"}  # 超时默认拒绝
```
`operations.py` 的 approve 端点里，L4/L5 需累计审批人（校验第二个审批人角色不同），并在 `resolve` 时把 `actor` 追加进 `approvals`；execute/reject 两节点都调用 `get_audit_logger().log(...)` 落 ES。超时后触发升级通知（复用 `_push_approval_card` 或 notifier）。

> 注意 `langgraph==0.2.28` 不支持 `interrupt()`（见进度说明技术债），故先用轮询；升级 langgraph 后再改为原生 interrupt/resume。

**验证**：提交 L4 事件 → 收到审批卡 → 单人批不通过、双人批才执行 → 不批到点自动 timeout → ES 审计可查。

### 修复 11（P2）：脱敏优先级 bug + 密码残留

**根因 A**：`mask.py:57-71` 的 `resolve_spans` 先按 `(-priority, start)` 排序，随后 `sorted(..., key=lambda s: s.start)` 又只按 start 重排，把优先级信息丢了 → 变成“起始最早者胜”，违反 credential>pii>hash。
**根因 B**：`default_rules.yaml` 密码正则 `\S+` 遇空格即停，`password = my secret` 只掩掉 `my`。

**正确改法 A**——一次排序内解决优先级+长度，冲突时保留高优先级/更长的：
```python
def resolve_spans(spans: list[Span]) -> list[Span]:
    if not spans:
        return []
    # 高优先级优先；同优先级更长优先；再按起点
    ordered = sorted(spans, key=lambda s: (-s.rule.priority, -(s.end - s.start), s.start))
    resolved: list[Span] = []
    for span in ordered:
        if not any(span.start < r.end and r.start < span.end for r in resolved):  # 无重叠才收
            resolved.append(span)
    return sorted(resolved, key=lambda s: s.start)
```
**正确改法 B**——密码正则改为匹配到行尾（或到分隔符），覆盖带空格的值：
```yaml
  - name: password_field
    pattern: '(?i)(?:password|passwd|pwd|secret)\s*[=:]\s*.+?(?=\s{2,}|[;,]|$)'
```
并在引擎里加脱敏后自检（文档 F1-3 要求“无明文密码残留”）：masked 文本再跑一遍密码正则，若仍命中则记 warning。

**验证**：新增用例——重叠的 credential+pii（credential 应胜）、`password = my secret pass`（整段应变 `***`）。
