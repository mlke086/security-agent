## 一、总体结论

项目已完成**阶段零（工程基建）+ 阶段一/二部分模块的骨架代码**，处于"骨架已搭、血肉待填"状态。

- 代码骨架完整（目录结构、配置、CI、docker-compose 均在位），但核心业务逻辑多为**单文件浅层实现**，核心代码合计约 557 行。
- **主图与真实子图尚未打通**：`main_graph/graph.py` 仍在用桩函数（`investigation_stub`、`vuln_hunter_stub`），而已写好的研判/漏洞/响应子图并未被主图挂载。
- **依赖未安装、代码未实跑**：当前环境 `langgraph` 等依赖尚未安装，没有证据表明任一模块被真实运行/测试过（仅 1 个单元测试文件）。
- **中间件已由运维实际部署**（见 `服务快速调用表.md`），但 `settings.py` 默认值仍指向 `localhost`，且实际环境新增了 Nacos / ClickHouse / MinIO 等原计划未覆盖的组件，需要对齐。

一句话：**里程碑 M0（预处理引擎就绪）和 M1（编排基线通）尚未真正达成验收标准**，需要先补齐"能跑起来、能验证"这一步。

## 三、关键风险与阻塞项（需优先处理）

1. **主图未挂真实子图（最高优先）**——整个系统当前无法端到端跑通任何真实流程，M1 无法验收。
2. **依赖未安装、代码从未实跑**——大量文件可能存在导入错误/语法问题却未被发现，"看起来完成"与"实际可运行"之间差距未知。
3. **配置与实际中间件不一致**——`settings.py` 默认 `localhost`/`changeme`，实际环境在 `192.168.80.101`，且密码为 `neo4j_password_2026` / `redis_password_2026`。不对齐则连不上任何中间件。
4. **实际环境新增组件未纳入设计**——`服务快速调用表.md` 出现 **Nacos（配置中心/注册）、ClickHouse（列存/分析）、MinIO**，原计划文档均未涉及。需明确：Nacos 是否用于配置下发？ClickHouse 是否替代/补充 ES 做事件分析存储？这会影响 settings、审计、指标模块设计。
5. **工具箱与记忆管理器为空**——研判/响应子图强依赖这两者，属于隐性阻塞。
6. **知识库为空**——GraphRAG 无数据，研判召回无从谈起。

## 四、下一步工作计划

按"先跑通、再补全、后优化"的顺序，分为四个批次（Sprint）。当前时点（7/10）对应原计划 Week 1，进度略滞后于原排期，建议先集中打通 M0/M1。

### Sprint 0：环境对齐与冒烟跑通（1-2 天，最高优先）

目标：让现有代码在真实中间件上"能装、能导入、能跑一条假事件"。

- [ ] **S0-1** 安装依赖：`pip install -e ".[dev]"`，逐一修复导入/语法错误，确保 `import src...` 全通过。
- [ ] **S0-2** 补 `.gitignore`（`.env`、`__pycache__`、`.venv`、`*.egg-info`、`node_modules` 等）。
- [ ] **S0-3** 对齐配置：更新 `settings.py` 默认值与 `.env.example`，指向实际中间件：
  - Milvus `192.168.80.101:19530`；Neo4j `bolt://192.168.80.101:7687` / `neo4j_password_2026`；
  - Kafka `192.168.80.101:9092`；ES `http://192.168.80.101:9200`；Redis `redis://:redis_password_2026@192.168.80.101:6379/0`。
- [ ] **S0-4** 写连通性自检脚本（替代/增强 `healthcheck.sh`）：用 Python 客户端分别 ping 通 Milvus/Neo4j/Kafka/ES/Redis，输出成功/失败清单。
- [ ] **S0-5** 决策项（需与运维/负责人确认）：Nacos、ClickHouse、MinIO 的定位。给出建议：MinIO 为 Milvus 底层存储（无需应用直连）；ClickHouse 建议用于运营指标/事件分析存储；Nacos 是否引入需评估，原型阶段可暂用 `.env`。
- [ ] **S0-6** 用 FastAPI 提交一条测试事件，跑通"入口→调度→桩子图→聚合"，确认主图 compile 与执行无误。

**验收**：`pytest` 可运行；`python -m src.api.main` 起服务；提交事件返回研判结论（桩）。

### Sprint 1：打通端到端主链路（对应 M1，3-5 天）

目标：主图挂真实子图，接通知识层与工具层最小可用版本。

- [ ] **S1-1** 实现 **MemoryManager**（`orchestration/memory/manager.py`）：Milvus+Neo4j 双写、证据去重、TTL。
- [ ] **S1-2** 实现 **工具箱最小集**（`knowledge/tools/`）：registry 装饰器 + VirusTotal + OTX + 通知（企业微信/钉钉）。先保证可 mock、可真调。
- [ ] **S1-3** **主图挂载真实子图**：用 `build_investigation_subgraph()` 等替换桩，配置入口映射与子图超时（研判120s/漏洞600s/响应300s）+ 错误回传。
- [ ] **S1-4** 知识库入库脚本：`scripts/import_attack_stix.py`（MITRE ATT&CK→Neo4j）、`scripts/ingest_knowledge.py`（CVE/情报→Milvus，含 BGE 嵌入）。
- [ ] **S1-5** GraphRAG 端到端联调：一条含 IOC 的事件跑通"检索→RRF融合→研判输出"。

**验收（M1）**：端到端跑通一条真实告警（非桩）；路由三分支均可达；审计日志全路径写入；GraphRAG 初步召回可评估。

### Sprint 2：补全预处理/研判验收 + 执行层（对应 M0/M2，5-7 天）

- [ ] **S2-1** 补单元测试至覆盖率 ≥90%：IOC 提取器、PoC Linter、路由函数、调度分类（mock LLM）。
- [ ] **S2-2** 预处理压测：Kafka 注入脚本，验证 ≥2000 events/s、P99<50ms、PII 召回≥99%、IOC 准确率≥98%（**补齐 M0**）。
- [ ] **S2-3** 沙箱补全：`deployments/docker/Dockerfile.sandbox` 最小镜像；创建 `sandbox-net`；SecComp + 资源限制 + 超时销毁；Linter P99<5ms 压测。
- [ ] **S2-4** 研判子图集成测试：50 条带标注事件，准确率≥80%（testcontainers 起真实 Milvus/Neo4j）。

**验收（M2）**：研判准确率≥80%；子图状态隔离验证；Linter/沙箱基础功能通过。

### Sprint 3：漏洞子图 + 响应子图 + API/前端（对应 M3/M4，按原 Week 6-11 推进）

- [ ] **S3-1** 漏洞挖掘子图闭环实跑：generate→linter→sandbox→extract→update→converge，防遗忘（第8-10轮约束保留）验证；PoC 成功率≥60%。
- [ ] **S3-2** 响应子图：编写 **10 类剧本 YAML**（`orchestration/playbooks/*.yaml`）；HITL `interrupt()`+Webhook 回调+Celery 超时任务实跑。
- [ ] **S3-3** FastAPI 补全：审批回调、推理轨迹、运营指标端点；OAuth2+JWT；RBAC 四角色。
- [ ] **S3-4** 前端初始化：React+Ant Design，4 页面（事件队列/推理轨迹/审批列表/运营大屏）。
- [ ] **S3-5** 5 类 E2E 联调场景。

**验收（M3/M4）**：漏洞验证闭环 + 沙箱0逃逸；审批流打通；5 场景通过；前端 4 页面可用。

### Sprint 4：生产加固（对应 M5）

- [ ] 压测（≥20000 events/s 扩展后、端到端 P99<30s）、72h 稳定性、安全审计（SAST/依赖/Prompt注入/RBAC越权/沙箱逃逸）、K8s 编排与部署文档、灰度发布。
