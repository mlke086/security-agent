# 计划 v2 — 对齐《安全AI_Agent项目开发文档 v1.1》的差距分析与开发计划

> 编制日期：2026-07-13
> 基准文档：《安全AI_Agent项目开发文档.md》（设计纲领 v1.1）
> 参照：项目审查V1.md / 项目审查V2.md（代码级复核结论）
> 目的：量化"当前代码 vs 设计文档"的差距，给出后续要做的事与优先级

## 一、总体差距结论

设计文档描述的是一套**从原型到生产**的完整系统。当前代码把四层架构的**骨架和大部分单点模块**都搭起来了（目录结构与文档第9.1节高度一致），但三大子图的**端到端闭环、生产级安全/合规、性能达标、前端与测试**普遍缺失。

| 维度 | 完成度（对设计文档） | 说明 |
|------|------|------|
| 架构骨架 / 目录结构 | ~90% | 与文档 9.1 基本一致，仅缺 `graphrag/fusion/` 独立目录（RRF 内联在 engine.py） |
| 数据预处理层（第3章） | ~70% | 脱敏/IOC/Kafka 都有，但有正确性 bug、无 NLP、无背压、性能未验证 |
| 编排与智能体层（第4章） | ~55% | 主图通；研判子图通；漏洞子图基本通；**响应子图未接入主图**；**置信度路由缺失**；记忆管理器未被有效使用 |
| 知识与工具层（第5章） | ~45% | RRF/Milvus/Neo4j 客户端有，但**嵌入模型未集成（用零向量）**、**Milvus 无索引**、安全设备工具（防火墙/EDR/SIEM）未实现 |
| 执行与隔离层（第6章） | ~50% | Linter 三通道有；Docker 沙箱能跑；但**无 gVisor/MicroVM**、**网络未按"仅靶机互通"隔离**、无容器池/编译环境/崩溃采集、HITL 未闭环 |
| API 层（第10.2节） | ~50% | JWT/RBAC/登录有；端点路径与文档不一致，`/kb/search`、`/approvals/{id}` 缺失，多为内存桩 |
| 前端（文档 2.2.1 用户交互层） | 待核实 | 文档要求 4 页面 React+AntD；`frontend/` 由脚本生成，需核实完整度 |
| 安全设计（第10.3节） | ~15% | 无 TLS/mTLS/KMS、无 Prompt 注入防护、无输出过滤、审计非不可篡改、密钥有泄露风险 |
| 测试策略（第11章） | ~25% | 67 个单测，覆盖率约 26%；无集成/E2E/混沌/专项安全/性能基准 |
| CI/CD（第9.4节） | ~10% | 有 `.pre-commit-config.yaml`；无 `.github/workflows`，无 Bandit/Safety |

**一句话**：原型级"能单点跑"，距离文档定义的"端到端闭环 + 生产加固"约完成一半，缺口集中在**闭环打通、安全合规、性能验证、测试与前端**。

## 二、逐层差距明细与待办

### 2.1 第3章 数据源与预处理层

已实现：脱敏引擎（YAML 规则+watchdog 热更新）、IOC 提取（IP/域名/哈希/URL+私网过滤）、aiokafka 消费+DLQ、标准 JSON Schema。

差距 / 待办：
- [ ] **修脱敏优先级 bug**：`mask.py:resolve_spans` 先按 `(-priority,start)` 排序又按 `start` 重排，丢弃了优先级；导致 credential>pii>hash 覆盖规则失效。
- [ ] **修密码残留**：`default_rules.yaml` 密码正则 `\S+` 遇空格即停，`password = a b` 会漏掉后半段；补脱敏后自检（确保无明文密码残留）。
- [ ] **NLP 脱敏（文档 3.2.1）**：文档要求"正则+轻量级NLP（BERT NER）"，当前仅正则。→ 集成一个轻量 NER（可选，生产项）。
- [ ] **规则引擎补 IP/域名规则**：文档 3.2.1 把 IP/域名列为规则引擎职责，当前硬编码在 extractor，YAML 无对应规则。
- [ ] **Kafka 背压/Consumer Lag 告警（计划要求）**：当前无 lag 监控。
- [ ] **吞吐优化**：`async for` 逐条 commit 抵消了 `max_poll_records=500`；正则在事件循环内同步跑，无 multiprocessing。→ 改批量消费 + CPU 卸载。
- [ ] **性能验证（M0 验收）**：P99<50ms、单实例≥2000 events/s、PII 召回≥99%、IOC 准确率≥98% 均无评测数据。→ 建评测集 + 压测脚本（现有 stress 脚本只测 producer 发送速率）。

### 2.2 第4章 编排与智能体层

已实现：主图 entry→orchestrator→(investigate|vuln_check|ignore)→aggregator；调度中枢（蜜罐规则+LLM few-shot）；研判子图（CTI→Investigator）；漏洞子图（generate→linter→sandbox→收敛）；MemoryManager（Milvus+Neo4j 双写+去重）。

差距 / 待办：
- [ ] **响应子图接入主图（P0）**：`main_graph/graph.py` 无 responder 节点，`route_decision` 只有 investigate/vuln_check/ignore。文档 4.5 与时序图 Step 14-18 的响应闭环在运行时完全不可达。
- [ ] **置信度阈值路由（P0）**：文档 4.3.2 明确 ≥0.8→响应 / 0.5-0.8→漏洞 / <0.5→归档。当前 confidence 记录但不路由，研判后直接 aggregator→END。
- [ ] **聚合节点真实聚合**：`aggregator` 只读 `subgraph_result.final_verdict/confidence`，漏洞子图成功返回的是 `final_poc/is_vulnerable/exploit_chain`，字段错配导致漏洞结论无法进入最终结论。
- [ ] **7维记忆对齐文档（4.4.1）**：文档 7 维为 target/code_paths/input_formats/poc_candidates/negative_evidence/verification_state/constraints。当前缺 `input_formats`、`verification_state`，被 `iteration_count/final_poc` 顶替。
- [ ] **防遗忘补全**：`poc_candidates` 从不写入 → `<memory_history>` 无历史 PoC 快照；缺 `<current_reasoning>` XML 标签。
- [ ] **收敛条件"路径耗尽"**：文档要求"成功 / 最大轮次 / 路径耗尽"三条件，当前只有前两条（`code_paths` 从未被读）。
- [ ] **记忆管理器落地**：`store_evidence` 仅 cti_analyst 调用；investigator/vuln_hunter/responder 未接。TTL 为 72h（文档 10.3.2 要求 24h）；`cleanup()` 扫描的 key 从未写入且不清 Milvus/Neo4j（无效）；`_store_in_graph` 每次新建 driver。
- [ ] **子图返回字段裁剪核对（文档 4.1.2 / 10.1）**：研判子图应仅回 final_verdict/confidence_score/evidence_summary（当前多回 mitre_ttps）；响应子图应回 approval_id/execution_summary（当前回 execution_result）。

### 2.3 第5章 知识与工具层

已实现：RRF 融合（k=60）、Milvus 客户端（含 ≥0.65 阈值过滤）、Neo4j 客户端（apoc N 跳）、Redis 缓存、Tool Registry 装饰器、VirusTotal/OTX/notifier 工具文件、模型适配层（Claude/OpenAI/vLLM）。

差距 / 待办：
- [ ] **嵌入模型集成（P0，文档 5.1.1）**：文档要求 BGE-large-zh-v1.5 生成 768 维向量；当前 `cti_analyst._query_graphrag` 用 `[0.0]*768` 零向量查询 → 向量检索这一路等于空跑。
- [ ] **Milvus 建索引**：`_ensure_collection` 建了集合但**未建索引**，`search` 里 `collection.load()` 在无索引时会失败。需 `create_index(IVF_FLAT)`。
- [ ] **知识入库（文档 5.1.1 / M2）**：MITRE ATT&CK、近3年 NVD CVE(CVSS≥7)、OTX 历史情报、内部应急报告入库。`scripts/import_attack_stix.py`、`ingest_knowledge.py` 存在，需核实是否真正跑通并有数据。
- [ ] **安全设备工具（文档 5.2）**：防火墙/SIEM/EDR/漏洞扫描器策略下发工具未实现（响应子图执行时只 log 不真正调用）。
- [ ] **微步在线**：文档列为 VirusTotal 同类，可选补充。
- [ ] **相似度阈值/兜底（计划 3.2.3）**：<0.65 过滤已在 Milvus 侧；外部情报 API 兜底逻辑需在 engine 层补。
- [ ] **模型适配层修正**：`chat_completion` 丢弃 system message、`temperature` 入参未透传；Claude 模型名硬编码 `claude-sonnet-4-5`。

### 2.4 第6章 执行与隔离层

已实现：PoC Linter 三通道（AST 语法/导入白名单/危险调用）、Docker 沙箱（seccomp=runtime/default、read_only、mem 1g、cpu 2核、超时、no-new-privileges）。

差距 / 待办：
- [ ] **沙箱网络隔离（P1，文档 6.1）**：当前 `network=sandbox-net`，文档要求"仅与靶机互通、无法访问外网/宿主机"。需 internal 网络 + 无出网。
- [ ] **gVisor/MicroVM（生产项，文档 6.1）**：文档要求生产用 gVisor（runsc）或 Firecracker；当前标准 Docker，无内核级隔离。
- [ ] **容器池预热**：文档要求预热 5 个空闲容器、冷启动<125ms；当前每次新建容器。
- [ ] **四组件补全**：代码编译环境（gcc/clang）、靶机运行环境、崩溃日志采集器（ASAN/Core Dump）；`execution/harness/` 目前为空。
- [ ] **Linter 加固**：`import os as o; o.system()`、`getattr` 反射调用可绕过危险调用检测。
- [ ] **HITL 闭环（P0，文档 6.3）**：`_LEVEL_TIMEOUT` 定义未启用；无超时默认拒绝+升级；L4 双人、L5 双主管审批未实现；审批记录未写 ES；用内存队列替代 `interrupt()`（0.2.28 限制，需评估升级 LangGraph 或补持久化）。

### 2.5 第10章 API 与安全设计

已实现：JWT（HS256）、RBAC（4 角色枚举）、`/auth/login`、`/auth/me`、`/events`、`/events/{id}/approve`、`/events/{id}/trace`、`/metrics`。

差距 / 待办：
- [ ] **端点对齐文档 10.2**：文档定义 `POST /approvals/{id}`、`GET /dashboard/metrics`、`GET /kb/search`；当前路径不同且缺 `/kb/search`（知识库检索）。`GET /events/{id}` 为写死桩。
- [ ] **Refresh Token**：`api_refresh_token_expire_days` 有配置但登录未签发 refresh token。
- [ ] **RBAC 角色补全**：文档要求 管理员/安全专家/分析师/审计员；代码只 seed 了 admin/analyst/viewer 三个用户（无 responder/auditor 用户）。
- [ ] **审计不可篡改（文档 10.3.1）**：审计写 ES 但失败静默、非 append-only/WORM；且主图各节点未真正落 ES（仅审批端点落）。→ 全节点审计 + ES ILM/只追加。
- [ ] **传输/存储加密（文档 10.3.2）**：无 TLS 1.3、无 mTLS、无 AES-256-GCM/KMS。
- [ ] **模型安全（文档 10.3.3）**：无 Prompt 注入检测（文档 NFR 要求 100% 拦截）、无输出敏感信息过滤、无模型行为监控。
- [ ] **密钥治理（P0）**：无 `.gitignore`、`.env` 含真实 DeepSeek key、`api_secret_key` 默认值可用；文档 9.2 明确禁止硬编码敏感数据。

### 2.6 前端（文档 2.2.1 用户交互层 / 计划 F6-2）

- [ ] **核实 `frontend/` 完整度**：文档要求 React+AntD 4 页面（告警队列/推理轨迹/大屏态势/审批卡片）。由 `gen_frontend.py` 生成，需确认是否可构建、是否对接了后端 API。

### 2.7 第9章 工程实践 / CI-CD / 第11章 测试

- [ ] **CI/CD（文档 9.4）**：建 `.github/workflows/ci.yml`：Ruff→mypy(strict)→pytest(cov≥90%)→Bandit→Safety→Docker build。
- [ ] **测试金字塔（文档 11.1）**：补集成测试（testcontainers 起 Milvus/Neo4j）、E2E（5 场景）、混沌测试；覆盖率 26%→90%。
- [ ] **专项安全测试（文档 11.2）**：沙箱逃逸、Prompt 注入、RBAC 越权、数据泄露、DAST(OWASP ZAP)。
- [ ] **性能基准（文档 11.3）**：500 万/日吞吐、P99<30s、50 并发、72h 稳定性。

## 三、按优先级的开发计划

### Sprint A — 安全底线与闭环打通（P0，1-2 周）

目标：消除泄露风险 + 让"研判→（漏洞）→响应→审批"端到端可达。

1. 加 `.gitignore`（`.env`/`.venv/`/`__pycache__/`/`.coverage`/`*.egg-info/`）；轮换已暴露的 DeepSeek key；`api_secret_key` 去默认值、启动强校验；清理文档中的明文服务器凭据。
2. 响应子图接入主图 + 实现置信度阈值路由（≥0.8/0.5-0.8/<0.5）。
3. 修 aggregator 字段错配，使漏洞子图结论正确进入最终结论。
4. HITL 闭环：启用超时→默认拒绝→升级；L4/L5 多人审批；审批与全节点审计写 ES。
5. 修脱敏优先级 bug + 密码残留 + 脱敏后自检。

### Sprint B — 知识层可用 + 记忆闭环（P1，2-3 周）

目标：让 GraphRAG 真正检索、记忆真正跨子图共享。

1. 集成 BGE-large-zh-v1.5 嵌入，替换零向量；Milvus 建 IVF_FLAT 索引。
2. 跑通知识入库（ATT&CK STIX + NVD CVE），产出可检索数据。
3. MemoryManager 落地：各子图接 `store_evidence`；TTL 改 24h；修 `cleanup()`；复用 Neo4j driver。
4. 7 维记忆对齐（补 input_formats/verification_state）；回填 poc_candidates；补 `<current_reasoning>`。
5. 模型适配层修正（system 消息、temperature、模型名可配）。

### Sprint C — 执行层加固 + 工具落地（P1，2 周）

1. 沙箱网络改"仅靶机互通"（internal 网络）；容器池预热；补编译/靶机/崩溃采集组件。
2. Linter 加固（别名/反射绕过）。
3. 安全设备工具（防火墙/EDR/SIEM/扫描器）实现，响应子图真正下发。
4. Prompt 注入检测 + 输出敏感信息过滤（文档 10.3.3 / NFR）。

### Sprint D — API/前端/测试/CI（P2，2-3 周）

1. API 端点对齐文档（`/approvals/{id}`、`/dashboard/metrics`、`/kb/search`）；Refresh Token；角色补全。
2. 前端 4 页面核实并对接。
3. CI/CD 流水线 + Bandit/Safety；覆盖率提升至 90%。
4. 集成/E2E（5 场景）/专项安全测试。

### Sprint E — 生产加固（P2，与文档阶段五对齐，2 周）

1. 性能基准（吞吐/延迟/并发/72h 稳定性）。
2. gVisor 生产隔离方案。
3. TLS/mTLS/KMS、监控告警（Prometheus+Grafana）、K8s 部署与灰度。

## 五、风险提示

- **langgraph 0.2.28 不支持 interrupt()**：HITL 用内存队列替代，进程重启即丢失待审批。生产需升级 LangGraph 或引入持久化 checkpointer。
- **零向量 GraphRAG**：在嵌入模型接入前，研判质量指标（M2）无法真实评估，勿以当前结果判断召回率。
- **沙箱当前隔离等级不满足"0 逃逸"**：标准 Docker 共享内核，接入 gVisor 前不应执行不可信/未知 exploit 类 PoC。
