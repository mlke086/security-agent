# 安全 AI Agent 项目代码审查报告 V1

> 审查日期：2026-07-13
> 审查基准：《安全AI_Agent开发计划_v1.1》（对应 `安全AI_Agent开发计划_v1.1.md`，计划正文 v1.0）
> 审查范围：`src/` 全量源码、`scripts/`、`deployments/`、配置与文档

---

## 总体评价

按计划的 F0–F6 六大功能域和目录结构（第 3.3 节）来看，代码骨架高度吻合：主图 / 子图 / 知识层 / 执行层 / API 都在正确的位置。Sprint 0-2 的预处理与执行层基本成形，Sprint 3 的响应子图和 API 处于"搭起来但没接通"的状态。

主要问题集中在三类：

1. **安全隐患** —— 密钥泄露风险、JWT 默认密钥、审计未落实、沙箱隔离不足。
2. **功能未闭环** —— 响应子图未接入主图、置信度路由缺失、多处逻辑坏死。
3. **与计划不符的实现偏差** —— 脱敏优先级、Vuln-Hunter 记忆维度、CTI 数据源、Kafka 批处理等。

---

## 一、安全问题（最高优先级）

### 1. `.env` 真实密钥有泄露风险，且没有 `.gitignore`
- 仓库根目录**不存在 `.gitignore`**，而 `.env` 已存在且包含真实的 DeepSeek `OPENAI_API_KEY`。目前 `.env` 还是 untracked，但一次 `git add .` 就会把密钥提交进版本库。
- `进度说明.md:96` 明文写了服务器 SSH 密码和代理地址。
- **建议**：立即添加 `.gitignore`（含 `.env`、`.venv/`、`__pycache__/`、`.coverage`、`*.egg-info/`），轮换已暴露的 DeepSeek key，并把文档里的凭据挪到密钥管理（计划第 5.3 节要求接入 KMS）。

### 2. JWT 默认密钥可被绕过签发 token
- `src/common/config/settings.py:58` 的 `api_secret_key` 默认值为 `"change-this-secret-key"`（长度刚好通过 16 位 `min_length` 校验）。生产若忘记覆盖，任何人都能伪造合法 JWT。
- **建议**：去掉默认值，启动时强制校验，缺失即拒绝启动。

### 3. 计划的"不可篡改审计日志"未落实
- `src/common/audit/audit_logger.py:38-42` 写 ES 失败时静默吞掉（只打日志），且用普通 ES index，没有 WORM / append-only 保证。计划 F5-3 明确要求"不可篡改"。
- 更关键：计划 F2-1 要求"全节点审计日志（每个节点执行追加一条结构化审计记录）"，但主图各节点只把 `audit_log` 写进 LangGraph state，**从未调用 `AuditLogger` 落 ES**。只有审批端点（`src/api/routers/operations.py:41`）真正写了 ES。全链路审计实际是断的。

### 4. 沙箱隔离与计划有差距
- `src/execution/sandbox/executor.py:77` 使用 `network=self._network`（`sandbox-net`）而非计划 3.2.5 层 4 要求的"仅与靶机互通、无法访问外网"。当前配置下 PoC 容器可能具备出网能力，与"沙箱 0 逃逸 / 网络严格隔离"目标冲突。
- 计划要求的容器池预热（5 个空闲容器、冷启动 < 125ms）未实现，每次 `execute` 现起新容器。
- Linter 的 `_call_name`（`src/execution/linter/poc_linter.py:106-112`）只能识别 `Name` / `Attribute` 形式，`import os as o; o.system()`、`getattr(os,'system')` 这类别名 / 反射调用可绕过危险调用检测。

## 二、功能未闭环（与计划最大的偏离）

### 5. 响应子图完全没接进主图
- `src/orchestration/main_graph/graph.py:91-129`：`route_decision` 只路由 `investigate / vuln_check / ignore`，主图里根本没有 responder 节点。计划 F3-3 的整条响应链路在运行时不可达。

### 6. 置信度阈值路由缺失
- 计划 F3-1 的核心逻辑"≥0.8 → 响应子图 / 0.5-0.8 → 漏洞子图 / <0.5 → 归档"在代码里不存在。研判子图是直线 `cti_analyst → investigator → END`，主图 `investigate → aggregator → END`。confidence 记录了但从不用于路由。

### 7. 响应子图内部三处坏掉
- `src/orchestration/subgraphs/responder/playbook_matcher.py:53` 用 `playbooks.extend(yaml.safe_load(...))`，但每个 playbook YAML 顶层是 dict，`extend` 会遍历出 key 字符串列表，后续 `pb.get(...)` 对 str 调用会抛 `AttributeError`。应改为 `.append`。
- 即使修好，playbook YAML 里没有 `trigger` 块，而匹配器按 `trigger.verdict / confidence_min` 匹配 → 规则匹配永远落空，"规则优先"形同虚设。
- HITL 审批超时 / 默认拒绝 / 升级（`_LEVEL_TIMEOUT` 定义了但从不使用）、L4 双人审批、L5 双主管审批都未实现；`hitl_approval_node` 直接返回 pending，L3+ 操作永远被跳过。

### 8. Vuln-Hunter 记忆维度与计划不符 + 防遗忘半失效
- 计划 F3-2 的 7 维是 target / code_paths / input_formats / poc_candidates / negative_evidence / verification_state / constraints。实际 `src/orchestration/subgraphs/vuln_hunter/memory.py` 缺 `input_formats` 和 `verification_state`，换成了 `iteration_count` / `final_poc` 等记账字段。
- `src/orchestration/subgraphs/vuln_hunter/poc_generator.py:33-41` 有两个重名 `PoCOutput`，第一个（含 confidence / trigger_condition）是死代码。
- `poc_candidates` 从不被填充，导致 `<memory_history>` 里没有历史 PoC 快照 —— 计划三重防遗忘保障里的"历史回注"实际只剩负面证据一路。计划的 `<current_reasoning>` XML 标签也未用上。

### 9. MemoryManager 存在但没被任何子图调用
- investigation / vuln_hunter / responder 没有一个 import 或调用 `store_evidence`，跨子图共享记忆实际没通。
- TTL 为 72h 而非计划的 24h（`src/orchestration/memory/manager.py:25`）。
- `cleanup()` 扫的 `event:ts:*` key 从没被写过，且不清理 Milvus / Neo4j，等于无效。
- `_store_in_graph` 每次新建 driver 而不复用 `self._neo4j`。

## 三、其他与计划不符 / 待改进

- **脱敏优先级 bug（影响合规）**：`src/preprocessing/sanitization/mask.py:62` 先按 `(-priority, start)` 排序，`:66` 又只按 `start` 重排，把优先级信息丢掉。结果是"起始位置最早的 span 胜出"，而非计划要求的 "credential > pii > hash 优先覆盖"。docstring 说的"同优先级更长者胜"也未实现。
- **密码残留**：`src/preprocessing/rules/default_rules.yaml:6` 的 `\S+` 遇空格即停，`password = my secret` 只掩掉 `my`，违反 F1-3"无明文密码残留"，且引擎未做计划要求的脱敏后自检。
- **CTI 少查一个源且非并发**：`src/orchestration/subgraphs/investigation/cti_analyst.py` 只调 VirusTotal + GraphRAG，OTX 从没被查（F3-1 要求三源并发）；且 VT 与 GraphRAG 是先后 await，不是真并发。
- **Kafka 消费**：`max_poll_records=500` 但实际 `async for` 逐条处理且每条都 commit，批量优化被抵消；背压 / consumer lag 告警（F1-5）完全缺失；正则脱敏在事件循环里同步跑，没有计划 3.2.2 的 multiprocessing 卸载，2000 events/s 目标存疑。
- **GraphRAG 未做阈值过滤**：`src/knowledge/graphrag/engine.py` 没有计划 F4-1 / 3.2.3 的相似度阈值（≥0.7 / <0.65 过滤）和外部情报 API 兜底。
- **IOC 域名漏报**：域名正则用固定 TLD 白名单，`.dev / .app / .me` 等被静默丢弃，威胁 98% 准确率 NFR；且脱敏后的邮箱域名会被当 IOC 误提取。
- **缺失的工程项**：无 CI/CD（F0-3 的 Bandit / Safety / ruff / mypy 流水线）、无规则覆盖率评估脚本、无 PII 召回率 / IOC 准确率评测集、测试覆盖率仅 26%（计划要求单测 ≥90%）。前端（F6-2）和 E2E（6.4 五场景）按进度说明尚未开始。

## 五、结论

项目在架构与目录层面忠实还原了开发计划，预处理与执行层的骨架可用，但距离计划描述的"端到端闭环、可上生产"仍有明显差距：**响应链路未接通、置信度路由缺失、审计未真正落库、脱敏优先级存在合规级 bug、密钥有泄露风险**。建议先处理 P0 安全项，再补齐编排闭环与正确性 bug，最后完善工程化与测试。
