## 一、安全问题（P0）

**S1. 无 `.gitignore`，`.env` 含真实密钥**
仓库根目录无 `.gitignore`（已确认）。`.env` 已存在且含真实 DeepSeek `OPENAI_API_KEY`（`OPENAI_BASE_URL=https://api.deepseek.com/v1`）。一次 `git add .` 即泄露。`进度说明.md:96` 另有明文 SSH 口令与代理地址。
→ 立即补 `.gitignore`（`.env`/`.venv/`/`__pycache__/`/`.coverage`/`*.egg-info/`），轮换已暴露 key。

**S2. JWT 默认密钥**
`settings.py:58` `api_secret_key` 默认 `"change-this-secret-key"`（刚好过 16 位 min_length 校验）。生产若忘改，任何人可伪造合法 JWT（`jwt.py:66` HS256 签名）。计划第5.3节要求 KMS。
→ 去默认值，启动强校验，缺失即拒绝启动。

**S3. 全链路审计未落 ES（计划 F2-1 / F5-3 要求）**
`AuditLogger`（`audit_logger.py`）只在审批端点（`operations.py:41`）被调用。主图各节点（entry/orchestrator/aggregator/ignore）只把 `audit_log` 写进 LangGraph state（内存），**从不落 ES**。且 `audit_logger.py:38-42` 写失败静默吞掉、用普通 index，无 append-only/WORM，"不可篡改"未实现。

**S4. Linter 危险调用可被别名/反射绕过**
`poc_linter.py:106-112` 的 `_call_name` 只识别 `ast.Name`/`ast.Attribute`。`import os as o; o.system(...)`、`getattr(os,"system")(...)`、`__builtins__` 间接引用均可绕过 `DANGEROUS_CALLS`。这是"沙箱0逃逸"的第一道防线，需按调用目标解析而非名字字符串匹配。

**S5. 沙箱隔离与计划 3.2.5 不符**
`executor.py:77` 用 `network="sandbox-net"`，非计划要求的"仅与靶机互通、禁止出网"。若该网络非 `internal`，PoC 容器可出网。另：容器池预热（5个空闲、冷启动<125ms）未实现，每次 `execute` 现起容器；gVisor 生产方案未见。

## 二、功能未闭环（P1，与计划偏离最大）

**F1. 响应子图未接入主图**
`main_graph/graph.py:91-129`：`route_decision` 只路由 `investigate/vuln_check/ignore`，主图无 responder 节点。`build_responder_subgraph`（`responder/graph.py`）从未被主图引用 → 整条 F3-3 响应链路运行时不可达。

**F2. 置信度阈值路由缺失（计划 F3-1 核心）**
计划要求"≥0.8→响应子图 / 0.5-0.8→漏洞子图 / <0.5→归档"。实际研判子图是直线 `cti_analyst→investigator→END`（`investigation/graph.py:14-16`），主图 `investigate→aggregator→END`。`confidence_score` 被记录但从不用于路由。

**F3. HITL 审批形同摆设**
- `hitl_approval_node`（`hitl_handler.py:99-102`）对 L3+ 直接返回 `"pending"` 后，边无条件走到 `execute_response`（`responder/graph.py:20`）；而 `execute_response_node:107` 见 `!= "approved"` 即 skip。→ L3/L4/L5 操作**永远被跳过**，既不等待也不轮询。
- `_LEVEL_TIMEOUT`（`hitl_handler.py:13-17`）定义了但从未使用 → 超时/默认拒绝/升级未实现。
- L4 双人审批、L5 安全+运维双主管审批（计划 F3-3）未实现，L3/L4/L5 都只建单条审批。

**F4. 规则剧本匹配永远落空**
10 个 playbook YAML 顶层字段为 `playbook_id/description/max_level/operations`（见 `malware_detection.yaml`），**无 `trigger` 块**；而 `playbook_matcher.py:74-78` 按 `trigger.verdict`/`trigger.confidence_min` 匹配 → 永远 `matched=None`，每次都退回 LLM 生成。"规则优先、无 LLM 调用"的设计失效。

**F5. GraphRAG 向量检索用零向量（严重）**
`cti_analyst.py:44` `mock_embedding = [0.0]*768`，`engine.py:45` 拿这个零向量查 Milvus。**没有任何地方集成 BGE-large-zh-v1.5**（计划 F4-1 明确要求）。即向量检索这一路实际是无意义查询，GraphRAG 只有图谱一半在工作。

**F6. Milvus 集合缺索引，search 会失败**
`milvus_client.py:52-66` `_ensure_collection` 建集合后**未创建索引**（计划要求 IVF_FLAT），而 `search:22` 调 `collection.load()`。无索引时 load/search 会抛错。另 `metric_type="IP"`（内积）与 `>=0.65` 阈值语义、与计划的相似度阈值 0.7 需对齐。

**F7. MemoryManager 的 TTL 与清理失效**
- TTL 为 72h（`manager.py:25`），计划要求 24h。
- `cleanup()`（`manager.py:85-102`）扫描 `event:ts:*` 键，但 `store_evidence` 从不写此类键，只写 `dedup:*` → 清理恒为 no-op；且只删 Redis，不清 Milvus/Neo4j，陈旧向量/图节点永不回收。
- `_store_in_graph`（`manager.py:128-133`）每次调用新建 `AsyncGraphDatabase.driver`，不复用 `self._neo4j`，连接泄露风险。

## 三、其他偏差与改进点（P2）

**脱敏（合规相关）**
- **优先级覆盖 bug**：`mask.py:62` 先按 `(-priority,start)` 排序，`:66` 又只按 `start` 重排，丢弃优先级 → 实际是"起始最早者胜"，违反计划 "credential>pii>hash 优先覆盖"。docstring 承诺的"同优先级更长者胜"也未实现。
- **密码残留**：`default_rules.yaml:6` `\S+` 遇空格即停，`password = my secret` 只掩 `my`，违反 F1-3"无明文密码残留"；引擎无脱敏后自检。
- **掩码模板是摆设**：`apply_mask:43-52` 忽略 YAML 模板串，硬编码 `matched[:3]+"****"+matched[-4:]`，改 YAML 的 `{prefix3}` 无效果。

**IOC 提取**
- 域名正则用固定 TLD 白名单（`extractor.py:11`），`.dev/.app/.me` 及多数 ccTLD 被静默丢弃，威胁 98% 准确率 NFR。
- 脱敏后邮箱保留域名（`a***@example.com`），会被域名正则误提取为 IOC。

**CTI 情报（F3-1）**
- **OTX 从未被查**（`cti_analyst.py` 只有 VirusTotal + GraphRAG），计划要求三源并发。
- **非真并发**：VT 先 `asyncio.gather`（`:72`）后，GraphRAG 才 `await`（`:81`），两路串行。

**Vuln-Hunter（F3-2）**
- 7 维记忆缺 `input_formats`、`verification_state`（`memory.py`），被 `iteration_count`/`final_poc` 记账字段顶替。
- **`poc_candidates` 从不被填充**：`generate_poc_node:89-92` 只 `add_constraint`，从不 append PoC → `_build_memory_history:64-66` 的 `<iteration>` 历史块永远为空，"历史 PoC 回注"防遗忘失效，只剩 `<negative_paths>` 一路。
- 计划的 `<current_reasoning>` XML 标签缺失（模板用 `<memory_history>/<constraints>/<target>/<current_task>`）。
- 收敛条件只判"成功 or 迭代≥10"，"路径耗尽"未实现（`code_paths` 从不被读）。

**Kafka 消费（F1-5）**
- `max_poll_records=500` 但 `run()` 用 `async for` 逐条处理且每条 commit（`consumer.py:54-62`），批量优化被抵消。
- 背压 / consumer lag 告警完全缺失。
- 正则脱敏在事件循环内同步执行（`consumer.py:69`），无计划 3.2.2 的 multiprocessing 卸载，2000 events/s 目标存疑。

**GraphRAG 引擎层**
- `engine.py:38` 缓存键只含 `ioc_values`、忽略 `query_vector`；RRF 融合无 0.7 阈值过滤与外部 API 兜底（计划 F4-1）。
- `neo4j_client.py:11` 依赖 APOC（`apoc.path.spanningTree`），属外部插件依赖，未在依赖清单显式声明。

**模型适配层**
- `adapter.py:52` 只保留 `role=="user"` 消息，丢弃 system message；`chat_completion` 无 `temperature` 透传到底层（参数收了但没用）。

**工程/测试（F0）**
- 无 CI/CD 流水线（计划 F0-3 的 ruff/mypy/pytest/Bandit/Safety）。
- 无规则覆盖率评估脚本、无 PII 召回率/IOC 准确率评测集。
- 覆盖率约 26%（计划要求单测 ≥90%）；前端（F6-2）、E2E 五场景（6.4）按进度说明未做。

## 五、结论

架构骨架与计划高度吻合，预处理层和执行层的静态组件基本成形。但**三大子图的"闭环"普遍缺失**：响应子图未接线、置信度路由不存在、HITL 不真正挂起、规则匹配永远落空、向量检索用零向量。这些让系统当前更接近"各模块可单独跑通的脚手架"，而非计划设想的端到端自动化闭环。

建议按上表 P0→P1→P2 推进，优先堵住密钥泄露与沙箱/Linter 安全缺口，再打通主链路闭环，最后补脱敏正确性与工程化（CI/评测/覆盖率）。
