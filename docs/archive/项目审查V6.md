## 三、仍影响功能的问题

### P1 — 严重（影响核心功能可用性）

#### P1-1 ScanMonitorPage/EventDetailPage SSE cleanup被闭包吞没
**文件**: `frontend/src/pages/ScanMonitorPage.tsx:116`, `EventDetailPage.tsx:67`
**问题**: SSE cleanup `return () => {...}` 在 wireEs/wireSource 函数内部，不是 useEffect 的直接返回值。组件卸载时 EventSource 不会被关闭。虽然加了 `eventSourceRef`，但 wireEs 内部的 return 无效。
**影响**: 页面切换时 EventSource 泄漏，累积连接。配合后端 SSE 终态结束（已修）减轻了影响，但仍可能在扫描进行中切换页面时泄漏。
**修复**: 将 cleanup 移到 useEffect 顶层（与 DashboardPage/EventQueuePage 一致）。

#### P1-2 VulnListPage 筛选器不触发数据刷新
**文件**: `frontend/src/pages/VulnListPage.tsx:93-96`
**问题**: 严重等级和状态两个 Select 的 onChange 只设 state，不触发 fetchData()。无监听 filterSev/filterStatus 的 useEffect。
**影响**: 筛选器完全失效，用户选择筛选后页面无反应，必须手动点"刷新"。
**修复**: 添加 `useEffect(() => { fetchData() }, [filterSev, filterStatus])`。

### P2 — 一般（影响体验/部分场景）

#### P2-1 对话式扫描 intent 解析不提取 nuclei 参数
**文件**: `frontend/src/components/ChatScan.tsx:687-703`
**问题**: `parseIntentFromSources` 只提取 targets/modules/engine，丢弃 nuclei_severity/nuclei_tags 等。
**影响**: 即使 LLM 解析出 nuclei 参数，前端不展示。
**修复**: 在返回对象中补全 nuclei_* 字段。

#### P2-2 DashboardPage 时间线最多展示20条
**文件**: `frontend/src/pages/DashboardPage.tsx`
**问题**: timeline 数据全部展示（无分页），数据量大时页面过长。
**影响**: 体验问题，不影响功能。

#### P2-3 主机组删除后 hosts.group_name 残留
**文件**: `src/agents/store.py:270-279`
**问题**: delete_group 不清理 hosts.group_name，导致 legacy 孤儿组。
**影响**: 前端组列表出现"未纳管"标签的孤儿组。
**修复**: delete_group 时清空关联主机的 group_name。

#### P2-4 删除任务 Redis 缓存不清
**文件**: `src/api/routers/vulnscan.py:delete` + store
**问题**: 删任务后 GET 仍返回 Redis queue:status 缓存（200而非404）。
**影响**: 不影响任务列表（从ES查），仅直接GET已删任务时返回缓存。
**修复**: delete_task 时 `redis.delete(f"vulnscan:queue:status:{task_id}")`。

### P3 — 代码质量

#### P3-1 main.py 预存 ruff E402/I001
**文件**: `src/api/main.py:23-25`
**问题**: configure_logging() 后再 import，ruff 报错。预存问题。

#### P3-2 task_queue/* mypy strict 错误
**文件**: `src/orchestration/task_queue/*.py`
**问题**: redis-py/LangGraph 类型 stub 不兼容。预存问题，不影响功能。

#### P3-3 settings.py logger 使用 lazy import（循环导入workaround）
**文件**: `src/common/config/settings.py`
**问题**: load_nacos_settings() 内用 lazy import logger 避免循环导入。非理想但可接受。

## 四、架构评估

### 后端架构
- **分层清晰**: FastAPI路由 → 业务层(manager) → 存储层(store) → PG/ES/Redis
- **Nacos集成**: 启动加载+30s轮询热更新，env > Nacos > .env > 默认值优先级正确
- **SSE流**: vulnscan task stream已修复终态结束（防止连接泄漏），其他stream仍无限循环（metrics/events）
- **WS网关**: keepalive 30s + agent read deadline 90s 配合正确，但跨worker投递返回False问题未修
- **规则分发**: Ed25519签名+HMAC分离+Bearer下载+CA验证，链路完整

- **React+Antd**: 组件化清晰，但是4个SSE页面cleanup模式不统一
- **状态管理**: 用React hooks（useState/useEffect），无Redux，适合当前规模
- **@ant-design/charts**: DashboardPage/ScanReportPage已替换为antd原生组件
- **vite dev proxy**: SSE有缓冲问题（dev模式），生产需nginx proxy_buffering off

### Agent端（Go）
- **规则引擎**: matcher三种check.type正确，规则持久化+rules.json落盘
- **WS通信**: keepalive处理正确，read deadline 90s+keepalive 30s配合正确
- **编译部署**: 交叉编译linux/amd64，需注意SELinux上下文(restorecon)

## 五、Agent 端补充审查（审查代理发现）

审查代理深度扫描了 Go agent 代码，发现 18 个新问题（P0×3, P1×6, P2×9）：

### P0 — 致命

#### P0-A1 agent_upgrade 成功ack丢失 + 失败双重ack
**文件**: `agent/cmd/agent/main.go:221-233`
**问题**: `HandleUpgrade` 成功后调 `os.Exit(0)`，永远不会执行成功ack。失败时没 `return`，既发错误ack也发成功ack。
**影响**: 服务端无法知道升级是否成功。

#### P0-A2 scan_cancel context 被丢弃，取消功能完全失效
**文件**: `agent/internal/scan/engine.go:160`
**问题**: `context.WithCancel` 返回的 ctx 被 `_` 丢弃。`Execute()` 和 `Match()` 都不检查 `ctx.Done()`。
**影响**: 服务端无法中断正在运行的扫描。

#### P0-A3 HandleUpgrade 使用裸 http.Get（无认证/CA/超时）
**文件**: `agent/internal/updater/upgrade.go:41`
**问题**: 与 HandleRuleUpdate（Bearer+CA+60s超时）不同，HandleUpgrade 用裸 `http.Get`。
**影响**: 需要认证时二进制下载失败。

### P1 — 严重（新增）

#### P1-A1 config_update 不应用资源限制到 Monitor
**文件**: `agent/cmd/agent/main.go:236-258`
**影响**: 运行时资源限制更新完全不生效。

#### P1-A2 资源 Monitor 从未启动
**文件**: `agent/internal/scan/engine.go`, main.go:105
**影响**: `IsThrottling()` 始终返回 false，基于资源的节流完全失效。

#### P1-A3 nuclei 扫描不可取消
**文件**: `agent/internal/scan/engine.go:314-317`
**影响**: nuclei 扫描无法被服务端取消。

#### P1-A4 并发 handleMessage goroutine 数量未限定
**文件**: `agent/internal/comm/client.go:203`
**影响**: 极端条件下可能 goroutine 堆积。

#### P1-A5 注册信息占位值（hostname作IP, 常量作kernel）
**文件**: `agent/internal/enroll/enroll.go:79-90`
**影响**: 控制台显示主机名而非IP，内核版本为常量。
