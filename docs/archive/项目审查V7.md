## 三、Standards 轴

### S1 [阻断] 当前变更无法通过仓库规定的 mypy CI

**依据**：`pyproject.toml` 开启 strict mypy；`.github/workflows/ci.yml:21-22` 明确执行 `mypy src/`。

**证据**：`src/agents/store.py:263` 的嵌套 bool 查询被推断为与前一个 term 查询不兼容，实测：

```text
src\agents\store.py:263: error: Dict entry 0 has incompatible type
"str": "list[dict[str, dict[str, str]]]"; expected "str": "str" [dict-item]
Found 1 error in 1 file (checked 122 source files)
```

**影响**：远端 lint job 必然失败，当前 HEAD 不满足合并门禁。

### S2 [阻断] 单元测试并非离线隔离，当前结果不绿

**依据**：`.github/workflows/ci.yml:33-36` 在没有 PostgreSQL service 的 runner 上直接运行全部单测；`tests/conftest.py:18-26` 却仍初始化真实 PG。

**实测**：`304 passed, 27 failed, 6 skipped`。其中：

- 26 个失败来自 API 测试连接共享 PostgreSQL 并依赖固定密码 `admin123` 等，当前库中的用户状态与测试假设不一致。
- 1 个失败来自 `tests/unit/agents/test_manager.py:131` 未同步本次新增的 `exclude_decommissioned=True` 调用参数。

**影响**：测试结果受外部数据库状态影响；“本机曾全绿”不能证明提交可在干净 CI 中复现。

### S3 [判断项：Duplicated Code] 规则版本被连续设置两次

**位置**：`agent/cmd/agent/main.go:211-214`。

`client.SetRuleVersion(req.RuleVersion)` 连续出现两次，行为目前幂等，但说明该修复由两段重复补丁叠加，容易在后续演化时产生分叉。

### S4 [判断项：Divergent Change] `ChatScan.tsx` 承担过多变化原因

**位置**：`frontend/src/components/ChatScan.tsx`，当前约 700 行，本次净增超过 200 行。

同一组件同时管理会话列表、模型切换、标题轮询、统一聊天路由、消息渲染、扫描确认、Nuclei 高级参数和任务跳转。该结构已经让“展示消息”和“真正持久化的消息”走上两条不同调用链，见 F2。

**Standards 小结**：4 项发现；最严重的是 S1（mypy 必然阻断 CI），其次是 S2（测试不可复现）。

## 四、Spec 轴

### SP1 [P0] V6 0a/0c：Agent 升级仍无法成功

V6 要求：

> “0a. agent_upgrade ack 逻辑修复”
>
> “0c. HandleUpgrade 下载加认证/CA/超时”

当前存在四个连续阻断点：

1. `src/api/routers/agents.py:627-632` 下发 payload 只有 `version` 和 `download_url`，没有二进制 `signature`。
2. `agent/internal/updater/upgrade.go:76` 强制要求 `signature`，因此当前 API 下发的升级必定失败。
3. 内置下载端点 `src/api/routers/agents.py:304-328` 只接受 enrollment token；Agent 在 `upgrade.go:45-48` 发送的是已经注册后的 agent token，二者不兼容。
4. `upgrade.go:105-113` 忽略两个 `os.Rename` 错误并直接删除 `.old`、`os.Exit(0)`；成功 ack 仍到不了 `agent/cmd/agent/main.go:236`。若替换第二步失败，还有删除唯一可执行文件的风险。

**结论**：0a 未完成，0c 仅完成了 CA/timeout 客户端复用，端到端升级不可用。

### SP2 [P0] V6 0b：取消链路没有从 API 接到 Agent

V6 要求：

> “0b. scan_cancel context 接入执行流”

`src/api/routers/vulnscan.py:215-222` 只把任务状态改成 `failed`，没有调用 gateway，也没有构造 `scan_cancel`。全仓搜索没有其他服务端发送点。

即便手工下发命令，Agent 侧也只在模块开始前检查 context；`runNuclei()` 在 `agent/internal/scan/engine.go:327` 创建独立的 `context.Background()`，正在运行的采集、匹配和 Nuclei 都不能被当前 cancel context 中断。

**影响**：界面/API 返回“取消成功”后扫描仍继续，后续结果可能继续写入并与 `failed` 状态竞争。

### SP3 [P2] V6 第 3 项：对话式 Nuclei 参数仍未接通

V6 要求：

> “parseIntentFromSources 返回对象补全 nuclei_severity/tags/templates/timeout_sec”

当前：

- `src/agents/models.py:110-114` 的 `ScanIntent` 连 `engine` 都没有，也没有任何 `nuclei_*` 字段。
- `frontend/src/components/ChatScan.tsx:687-699` 的 `parseIntentFromSources` 仍只返回 targets/modules/engine/resource_limit/schedule。
- `IntentCard` 虽然渲染了 Nuclei 控件，但无法得到 LLM 已解析的 Nuclei 参数。

**结论**：该计划项未完成；对话式入口实际上总会回落到 matcher。

### SP4 [P2] V6 第 4 项：主机组关联清理仅部分处理

V6 要求：

> “delete_group 时 UPDATE hosts SET group_name=NULL WHERE group_name=$1”

`src/agents/store.py:320-333` 改成“存在非 decommissioned 成员则拒绝删除”，没有清理关联字段；decommissioned 主机不计数，因此组仍可被删除并留下 `hosts.group_name` 残留。

### SP5 [P2] V6 第 5 项：删除任务仍未清 Redis 状态缓存

V6 要求：

> “delete_task 时 redis.delete(f\"vulnscan:queue:status:{task_id}\")”

`src/api/routers/vulnscan.py:225-240` 与 `src/agents/store.py:479-495` 只删除 ES 数据。`api_get_task` 仍会回退读取 `vulnscan:queue:status:{task_id}`，所以已删除任务可能继续返回 200。

### SP6 [P1] V6 第 11 项：Nacos 62 键可读取，但“热更新验证通过”范围过窄

V6 声明：

> “62个配置项全部就位；热更新验证通过”

本次实测拉取 62 键成功，但：

- `src/api/main.py:101-112` 在加载 Nacos 前已经初始化 ES 与 PG，Nacos 中的核心连接配置无法支持冷启动。
- `settings.py:202-210` 仅在首次 fetch 非空时启动 poller；启动瞬间 Nacos 不可用时，进程此后不会自动恢复。
- poller 只重建 `Settings` 单例，不重建已经创建的 PG pool、ES/Redis 客户端等资源；删除 Nacos 键时旧的 `os.environ` 值也不会清除。

**结论**：配置拉取已完成，端到端动态生效和故障恢复未完成。

**Spec 小结**：6 项发现；最严重的是 SP1（升级链路不可用）与 SP2（取消链路未接通）。

## 五、功能性补充审查

### F1 [P1] “物理删除主机”按钮永远不会发送 `purge=true`

`HostOnboardPage` 调用 `deleteHost(agentId, true)`，但 `frontend/src/api/client.ts:126-128` 将第二个参数命名为 `_force` 后完全丢弃，只执行普通 DELETE。

**影响**：已下线主机点击“删除”后仍只是再次软下线，物理删除功能不可用。

### F2 [P1] 每条对话会调用两次模型，展示历史与持久化历史可能不同

`frontend/src/api/client.ts:331-346` 先调用统一 `/chat` 获得展示回复，再调用 `/vulnscan/conversations/{id}/chat`“持久化”。后一个接口会再次调用模型，并把第二份回复写进 PG。

此外，执行扫描时 `ChatScan.tsx:163-167` 又把全部用户消息拼成一个新消息调用同一接口。

**影响**：正常发送至少双倍模型成本和延迟；刷新会话后可能看到与刚才界面不同的回答；执行一次扫描还会污染对话历史。

### F3 [P1] `config_update` 返回成功，但配置没有真正动态生效

`agent/cmd/agent/main.go:253-261` 只修改 `cfg.HeartbeatSec` 并打印资源值：

- 心跳 ticker 已在 `agent/internal/comm/client.go:216` 按旧值创建，不会重建。
- resource limit 和 log level 没有应用，也没有持久化。
- 默认保护参数为 5 秒 CPU 采样、1 秒 poll timeout（`protection.go:44-45`），CPU 探测会先超时；旧 `resource.Monitor` 又未启动。

**影响**：控制台收到成功 ack，但心跳周期和资源限制实际不变，主机自保护不可信。

### F4 [P1] 主机“下线”不撤销凭证，也不关闭现有 WebSocket

`src/agents/manager.py:111-117` 只更新状态并删 Redis key，没有更新 `agent_tokens.revoked_at`，也没有断开 `_conns` 中的连接。

**影响**：下线 Agent 仍可能保持连接并接收直接命令；重连时 `register_online()` 会再次把它标成 online。当前“下线”不是有效的控制或安全边界。

### F5 [P1] 跨 worker 命令已发布却固定返回失败

`src/agents/ws_gateway.py:218-224` 成功 `redis.publish()` 后仍固定 `return False`。调用方据此返回 404 或统计失败，而订阅 worker 可能已经把命令发给 Agent。

**影响**：用户看到失败并重试时可能产生重复扫描、重复升级或重复配置命令。该问题在 V6 报告中已知，但本轮仍未修复。

### F6 [P2] Nuclei 结果被静默改写成 `sys_vuln`

Agent 的 Nuclei runner 产生 `category="nuclei"`，但 `src/agents/ws_gateway.py:335-337` 只允许 `sys_vuln`/`baseline`，其他值一律改成 `sys_vuln`。

**影响**：报告、筛选和统计无法区分 Nuclei 发现与系统包漏洞，审计语义丢失。
