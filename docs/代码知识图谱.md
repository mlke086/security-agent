## 4. 主图数据流（state 字段流转）

`run_pipeline` 播种 `{event_id, raw_event, audit_log:[]}`：

| 节点 | 读 | 写 |
|---|---|---|
| `entry` | raw_event.event_id | event_id, stage="triage", audit_log |
| `orchestrator` | raw_event | priority, event_tags, stage="route", audit_log（蜜罐规则+LLM分类，异常回退 medium） |
| `route_decision` | priority, event_tags | → investigate / vuln_check / ignore |
| `investigate`（包 investigation 子图） | raw_event.iocs | subgraph_result={final_verdict, confidence_score, evidence_summary, mitre_ttps} |
| `vuln_check`（包 vuln_hunter 子图） | raw_event | stage="verify", subgraph_result={final_poc, is_vulnerable, exploit_chain}（**无 final_verdict/confidence**） |
| `aggregator` | subgraph_result | final_verdict, confidence_score, **stage="done"（覆盖！）**, audit_log |
| `route_after_verdict` | subgraph_result, stage | → respond / vuln_check / done |
| `respond`（包 responder 子图） | subgraph_result | pending_action={approval_id, execution_summary} |
| `ignore` | — | final_verdict="ignored", stage="done", audit_log |

> ⚠️ `aggregator` 把 stage 强制改成 "done"，使 `route_after_verdict` 的 `stage=="verify"` 分支永远走不到 → vuln_check→respond 链路死。详见审查V3 [P1-CORE-1]。

`runner.run_pipeline` 收尾：把 `audit_log` 写成 EventStore trace（**action 恒为 "processed"**），按有无 `approval_id` 设 status=pending_approval/completed。

## 5. 各子图内部调用图

### 5.1 investigation
```
START -> cti_analyst -> investigator -> END   (无分支)
  cti_analyst: 并行 VT(最多5) + GraphRAG(embed->milvus+neo4j->RRF) + LLM(IntelCard)
               -> store_evidence 到 MemoryManager
  investigator: LLM CoT -> VerdictResult(verdict/confidence/evidence/mitre)
```

### 5.2 vuln_hunter（PoC 验证，≤10 轮）
```
START -> generate_poc -> linter_check -> sandbox_exec -> check_convergence
                                  │                       │
                                  └─(linter_fail 标记)─►   ├─► generate_poc (循环)
                                                          └─► finalize -> END
  generate_poc: LLM 生成 PoC；iteration_count++；写 memory.constraints
  linter_check: PoCLinter.check（语法/导入/危险调用）
  sandbox_exec: SandboxExecutor.execute（Docker 隔离，stdout 找 "VULNERABLE"）
  check_convergence: is_vulnerable OR iteration>=10 -> finalize
```

### 5.3 responder
```
START -> playbook_matcher -> hitl_approval -> route_after_hitl -> {execute_response | reject} -> END
  playbook_matcher: 规则优先(trigger.verdict/confidence_min/event_tags) -> 命中即用；未命中 LLM 生成
  hitl_approval: L1/L2 自动批；L3+ 建审批行+企微/钉钉卡片+wait_result(阻塞到终态)
                 _LEVEL_TIMEOUT: L3=300/L4=900/L5=1800s；_REQUIRED_APPROVERS: L3=1/L4=2/L5=2
  execute_response: ActionDispatcher.execute_playbook（幂等+dry-run+rollback）
```

### 5.4 vulnscan（主机漏扫，独立子图）
```
START -> parse_intent -> dispatch -> collect -> aggregate -> llm_analysis -> generate_report -> END
  parse_intent: LLM 解析扫描意图(source/targets/modules)
  dispatch: 建 ScanTask(ES) + 解析目标 + 广播签名 scan_command 到 agent
  collect: 读 ES scan_result（★线性图，无法等待 agent，详见审查V3 [P0-VS-2]）
  aggregate: 去重 + 批量存 vuln
  llm_analysis: 批量 LLM 过滤/定级/修复建议
  generate_report: 统计 + AI 摘要 + 发布 done
```

## 7. 关键集成缝隙（改 bug 前必读）

1. **vulnscan ⊥ main_graph**：两条线无连接。若要让"主图研判→触发漏扫"，需在主图加边，目前没有。
2. **WS 认证断裂**：`ws_gateway.authenticate` 读的 Redis key 从无写入，`validate_agent_token`(PG) 未接线 → agent 连不上。
3. **签名密钥断裂**：`signing._get_private_key` 在 key 为空时每次随机生成 → Go agent 永远验签失败（且 Go 侧公钥也从未下发）。
4. **collect 无法等待**：vulnscan 子图是线性的，dispatch 后立即 collect，agent 还没扫完 → 报告恒空。
5. **审计 trace 不完整**：investigate/vuln_check/respond 不往 `state["audit_log"]` 追加 → runner 写出的 trace 缺这几步。
6. **沙箱+linter 安全叠加**：linter 可被 `sys.modules["os"]` 绕过 + 沙箱非 internal + 无 gVisor → PoC 可逃逸出网。
