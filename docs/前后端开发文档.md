### 1.2 端到端处理流程（现状）

```
告警文本
  │  (POST /api/v1/events)
  ▼
entry ──► orchestrator ──分流──► investigate ──► aggregator ──阈值路由──► respond (HITL) ──► END
   (校验)    (LLM 定级)     │                    (汇总 verdict)      │
                           ├──► vuln_check ──────────┘                └──► done (归档)
                           └──► ignore (噪音)
```

- **分流规则**：`priority=low → ignore`；`tags 含 vulnerability → vuln_check`；`high/medium → investigate`
- **阈值路由**：`verdict=true_positive 且 conf≥0.8 → respond`；`0.5≤conf<0.8 → vuln_check`；否则归档
- **HITL**：操作分级 L1–L5，L1/L2 自动通过，L3+ 进入待审批队列，超时/多人会签

### 1.3 已有前端现状

已初始化 **React 18 + TypeScript + Vite + Ant Design 5**，含 4 个页面骨架与真实 API 调用，但存在明显缺口：

| 页面 | 现状 | 缺口 |
|------|------|------|
| 登录 LoginPage | ✅ 可用 | — |
| 运营大屏 DashboardPage | ⚠️ 4 个统计卡片 | 无图表、无自动刷新 |
| 事件队列 EventQueuePage | ❌ 仅提交表单 | **没有事件列表/队列表格**，结果以裸 JSON 展示 |
| 推理轨迹 TracePage | ⚠️ 只显示计数 | **不渲染推理步骤明细**，只显示 trace_count |
| 审批列表 `/approve` | ❌ 复用 TracePage | **没有待审批列表**，必须先知道 event_id |

### 1.4 演示视角下的关键阻塞点

要「在界面给别人展示项目功能」，当前最缺的是**可视化叙事能力**：

1. **后端无事件持久化与列表**：`submit_event` 处理完不落地，`_traces` 永远为空，前端无法列出「处理过哪些事件」。
2. **推理轨迹拿不到步骤明细**：`/trace` 依赖 ES 审计日志，演示环境未必有 ES；而图执行返回的 `audit_log`（每个节点的推理步骤）没有被 API 捕获返回。
3. **待审批队列不可枚举**：`hitl_handler` 没有 `list_pending_approvals()`；且审批队列在图执行进程内存中，独立 API 进程看不到。
4. **前端缺少「事件全景 + 单事件推理钻取 + 审批操作」的闭环页面**。

## 二、设计目标与原则

1. **演示优先，零外部依赖可跑**：Dashboard/事件列表/轨迹/审批四页在**不连 ES/Kafka/Neo4j** 的情况下也能演示（用进程内内存事件仓 + 可选 mock 数据）。
2. **复用已跑通的主图**：提交事件仍走真实 LangGraph 主链路，把每个节点的 `audit_log` 完整捕获并回显——这就是最有说服力的「AI 推理过程」。
3. **不破坏现有后端契约**：新增端点，保留旧端点。
4. **最小侵入**：后端新增一个 `EventStore`（内存单例），把散落的 `_approvals`/`_traces` 收敛进去；给 `hitl_handler` 补一个列表访问器。

### 3.2 数据模型（EventStore 记录）

```python
class TraceStep(BaseModel):
    node: str            # entry | orchestrator | investigate | vuln_check | aggregator | respond | ignore
    action: str          # 该节点做了什么
    summary: str         # 人类可读摘要
    timestamp: str
    details: dict        # 结构化细节（verdict/confidence/iocs/poc 等）

class EventRecord(BaseModel):
    event_id: str
    source: str
    submitted_at: str
    finished_at: str | None
    status: str          # processing | completed | pending_approval | ignored | error
    priority: str | None # low | medium | high | critical
    tags: list[str]
    sanitized_text: str
    iocs: dict[str, list[str]]
    final_verdict: str | None      # true_positive | false_positive | ignored | ...
    confidence: float | None
    evidence_summary: str | None
    mitre_ttps: list[str]
    poc: str | None                # 漏洞子图产出（如有）
    is_vulnerable: bool
    trace: list[TraceStep]
    approvals: list[dict]          # 审批历史
    pending_approval_id: str | None
    duration_ms: int | None
```

### 3.4 提交事件的异步化（关键改造）

现状 `submit_event` 同步 `await graph.ainvoke(...)`，跑一条真实链路可能几十秒（含 LLM 调用），前端会长时间转圈。改为：

1. 生成 `event_id`，写入 EventStore（`status=processing`），**立即返回**。
2. 用 FastAPI `BackgroundTasks` 或 asyncio 任务跑主图。
3. 图执行中/结束把 `audit_log`（每节点推理步骤）逐条写入该事件的 `trace`；结束时更新 `status/verdict/confidence/poc`。
4. 前端事件列表/详情页轮询（或后续升级 SSE/WebSocket）刷新状态。

> 演示建议：保留一个 `sync=true` 查询参数走同步模式，便于「点一下立刻看到完整结果」的现场演示。

### 3.5 审批队列跨进程问题

当前审批队列是图执行进程的模块级内存。若 API 与图执行同进程（演示默认单进程 uvicorn），`list_pending_approvals()` 即可工作。**演示部署务必单进程运行**（不要用多 worker），否则需要外置 Redis 存审批状态（列为后续项，非演示必需）。

| `/events` | 事件队列 | 全部 | 事件表格（状态/定级/结论/耗时）+ 筛选 + 「提交新事件」抽屉 + 「注入演示数据」按钮 |
| `/events/:id` | 事件详情/推理轨迹 | 全部 | 事件概览 + **推理链时间线（分节点渲染）** + 证据/MITRE/PoC + 审批历史 |

> 用 `/events/:id` 取代原来靠 query 参数的 TracePage，钻取更自然。

### 4.3 关键组件设计

**事件队列表格（EventQueuePage）**
- 列：提交时间、来源、定级 Tag（颜色区分）、结论 Tag、置信度进度条、状态徽标、耗时、操作（查看轨迹）。
- 顶部筛选：状态/结论/定级下拉。
- 右上角：`提交新事件`（Drawer 表单：描述、来源、IOC）、`注入演示数据`。
- `status=processing` 时行内显示 Spin，列表 5s 轮询。

**推理轨迹时间线（EventDetailPage）— 演示核心**
- 顶部 Descriptions：event_id、结论、置信度、定级、MITRE TTP 标签、耗时。
- 中部 AntD `Steps`/`Timeline` 逐节点渲染 `trace`：
  - `orchestrator`：展示 LLM 定级理由、priority、tags
  - `investigate`：情报卡摘要、verdict、置信度、证据
  - `vuln_check`：PoC 代码块（`Typography` 高亮）、沙箱执行结果、是否可利用
  - `respond`：匹配的 playbook、操作分级、审批状态
  - 每步带节点图标、耗时、可折叠详情（JSON 详情用 `Collapse`）。
- 底部：审批历史 Timeline；若 `pending_approval`，显示通过/驳回按钮（角色允许时）。

**运营大屏（DashboardPage）**
- 4 统计卡片：总事件、真阳性、待审批、平均耗时。
- 饼图：结论分布（true/false positive/ignored）。
- 柱图：定级分布（low/medium/high/critical）。
- 时间线/折线：最近事件处理量。
- 每 10s 自动刷新。

**审批列表（ApprovalsPage）**
- 表格：event_id、操作分级 Tag、已审批/需审批（如 1/2）、创建时间、状态、操作（通过/驳回 + 备注）。
- 操作后刷新，并可跳转 `/events/:id` 看上下文。
