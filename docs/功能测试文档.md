## 三、后端 API 用例

> 每条含：编号、路径、输入、期望、判定。可用 curl / Postman / Swagger（`/docs`）执行。

### 3.1 认证

| 编号 | 用例 | 输入 | 期望 |
|------|------|------|------|
| API-AUTH-01 | 正确登录 | `POST /auth/login {admin/admin123}` | 200，返回 `access_token`、`role=admin` |
| API-AUTH-02 | 错误密码 | `POST /auth/login {admin/wrong}` | 401 Invalid credentials |
| API-AUTH-03 | 携带 token 取自身 | `GET /auth/me` + Bearer | 200，`username=admin,role=admin` |
| API-AUTH-04 | 无 token 访问受保护端点 | `GET /metrics` 无 header | 401 Not authenticated |
| API-AUTH-05 | 过期/伪造 token | 篡改 token | 401 |

### 3.2 事件

| 编号 | 用例 | 输入 | 期望 |
|------|------|------|------|
| API-EVT-01 | 提交事件（异步） | `POST /events {sanitized_text, iocs, source}` | 200，`{event_id, status:"processing"}`，立即返回 |
| API-EVT-02 | 提交后可查详情 | `GET /events/{id}` | 200，随后 status 由 processing→completed，trace 非空 |
| API-EVT-03 | 事件列表 | `GET /events` | 200，`items` 含刚提交事件，`total≥1` |
| API-EVT-04 | 列表按状态筛选 | `GET /events?verdict=true_positive` | 200，仅返回真阳性事件 |
| API-EVT-05 | 分页 | `GET /events?limit=2&offset=0` | 200，最多 2 条 |
| API-EVT-06 | 详情含推理链 | `GET /events/{id}` | `trace` 数组含 entry/orchestrator/... 各节点步骤 |
| API-EVT-07 | 同步模式（演示用） | `POST /events?sync=true` | 200，返回时已 completed，trace 完整 |
| API-EVT-08 | viewer 不能提交 | analyst 之外角色 `POST /events` | 403 |

### 3.3 推理轨迹与审批

| 编号 | 用例 | 输入 | 期望 |
|------|------|------|------|
| API-TRC-01 | 取轨迹 | `GET /events/{id}/trace` | 200，`trace`、`approvals`、计数 |
| API-APR-01 | 待审批列表 | `GET /approvals` | 200，含 L3+ 事件的 pending 项 |
| API-APR-02 | 通过审批 | `POST /events/{id}/approve?action=approved` | 200，事件状态更新，approvals 追加记录 |
| API-APR-03 | 驳回审批 | `POST /events/{id}/approve?action=rejected&note=xxx` | 200，状态=rejected，note 记录 |
| API-APR-04 | analyst 不能审批 | analyst `POST .../approve` | 403 |
| API-APR-05 | 审批后队列移除 | 通过后再 `GET /approvals` | 该项不再 pending |

### 3.4 指标与演示数据

| 编号 | 用例 | 输入 | 期望 |
|------|------|------|------|
| API-MET-01 | 指标聚合 | `GET /metrics` | 200，含 total_events、各 verdict/priority 计数、待审批数、平均耗时 |
| API-MET-02 | 提交后指标变化 | 提交事件后再取 | total_events +1 |
| API-SEED-01 | 注入演示数据 | `POST /demo/seed` | 200，注入 5 类样例；列表/大屏立即有内容 |
| API-SEED-02 | 非 admin 注入 | analyst `POST /demo/seed` | 403 |
| API-HEALTH-01 | 健康检查 | `GET /health` | 200 `{status:ok}` |

## 五、端到端场景（5 类）

> 每类场景走完整链路，既是功能验证，也是演示脚本。

### E2E-01：蜜罐命中 → 研判 → 真阳性
- **输入**：`sanitized_text` 含蜜罐关键词，`iocs.ips=["203.0.113.5"]`，source=honeypot
- **预期链路**：entry → orchestrator(priority=high) → investigate → aggregator(true_positive, conf≥0.8) → respond
- **界面验证**：详情页可见 orchestrator 命中蜜罐规则、investigate 给出 verdict=true_positive、respond 生成 playbook
- **判定**：状态 completed 或 pending_approval，大屏真阳性 +1

### E2E-02：CVE → 漏洞挖掘 → PoC 验证
- **输入**：`sanitized_text` 描述某 CVE，`tags` 触发 vulnerability（或 orchestrator 打标）
- **预期链路**：orchestrator → vuln_check（生成 PoC→lint→沙箱→收敛）→ aggregator
- **界面验证**：详情页 vuln_check 步展示 PoC 代码块、沙箱执行结果、`is_vulnerable`
- **判定**：trace 含 vuln_check 步骤，PoC 非空

### E2E-03：噪音事件 → 忽略
- **输入**：低价值告警文本，无有效 IOC
- **预期链路**：orchestrator(priority=low) → ignore
- **界面验证**：详情页结论=ignored，链路短
- **判定**：状态 ignored，不进入审批

### E2E-04：高危事件 → 响应 → HITL 审批 → 执行
- **输入**：高危可处置事件（触发 L3+ 操作分级）
- **预期链路**：investigate(true_positive) → respond → hitl(pending) → 审批通过 → execute
- **界面验证**：审批列表出现待审批项 → responder 通过 → 详情页 respond 步显示已执行记录
- **判定**：审批前 status=pending_approval，通过后 completed，approvals 有记录

### E2E-05：认证 + 授权完整流程
- **输入**：分别用 4 个角色登录操作
- **预期**：
  - viewer 提交事件被拒（前端无入口/后端 403）
  - analyst 可提交、看大屏，审批按钮不可用
  - responder 可审批
  - admin 全权限 + 注入演示数据
- **判定**：权限矩阵（开发文档第五节）逐格符合
