## 三、本轮修复中遗留的工程化事项

### 1. venv312 损坏，影响本地单测 / mypy
- 现象：`.venv312/Lib/site-packages/` 下有 dist-info 但缺 `_pytest`、`librt`、`pathspec`、`six`、`mypy_extensions`、`typing_extensions` 等的 __init__.py。
- 影响：无法在本机跑 `pytest`、无法跑全量 `mypy`。
- **建议**：
  ```bash
  cd V:\project\security-agent
  rm -rf .venv312
  python -m venv .venv312
  .venv312\Scripts\python.exe -m pip install -r requirements.txt
  .venv312\Scripts\python.exe -m pip install pytest pytest-asyncio pytest-cov
  ```
- 跑回归：
  ```bash
  $env:PYTHONPATH = "V:\project\security-agent"
  $env:STORE_BACKEND = "memory"
  .venv312\Scripts\python.exe -m pytest -o addopts=""
  ```

### 2. crypto.PublicKey 并发安全
- 当前：`agent/internal/crypto/verify.go` 用包级 `var PublicKey`。
- 风险：多 goroutine 同时读 + SetPublicKey 调用会有 race。
- **建议**：改为 `atomic.Value` 或 `sync.RWMutex` 包裹。下次修 P1-GO-2 时一并处理。

### 3. P0-VS-2 顺带没收尾的 P2-API-12
- `src/api/routers/vulnscan.py:46` 的 `asyncio.create_task(...)` 仍 fire-and-forget。
- 已用 task_id 透传修了 P0-VS-2，但 task 失败时仍会静默（除非加 `try/except` + 持久化引用）。
- 下次顺手处理。

### 4. main.py `_ensure_es_indices` 仍可在 except 路径泄露 `es`
- 本次只移除了 `await vs.close()`，但 `AsyncElasticsearch(hosts=...)` 在 except 触发时仍不会 aclose。
- 建议下次加 try/finally + `await es.close()`。

### 5. 文档/索引同步
- 修复了 `vulnscan` router + subgraph 后，`docs/代码知识图谱.md` 的"6.2 漏扫子图"和"集成缝隙"节需要相应更新（标注 P0-VS-1/2/3/4/5/7 已修）。
- 建议：V4 修复完后再做一次 review，刷新两篇文档。

## 九、Linux snippet 末尾字面 `\n` 修复 (2026-07-17, 用户反馈)

**用户反馈**：

```
bash: secagent-install.shn: No such file or directory
```

**根因**：raw f-string `rf'...\n'` 里 `\n` 是**两个字面字符**（反斜杠 + n），不是真换行。粘到 bash 后 bash 看到文件名是 `secagent-install.sh\n`（多了反斜杠 n），找不到。

```python
# 错的：
snippet = rf'... secagent-install.sh\n'    # -> "... sh\n"
# 对的（任选其一）：
snippet = rf'... secagent-install.sh'        # 不加任何末尾
snippet = f'... secagent-install.sh\n'     # 真换行（但末尾换行通常不必要）
```

**修复**：
- `/install-helper` Linux 分支去掉末尾的 `\n`，结尾停在最后一个命令
- Windows 分支用普通 f-string（不是 raw），行间是真 `\n`、末尾不留空行
- 前端 Windows fallback 同样把 `\\n`（两个字面）改成 `\n`（真换行）
- 加 1 条回归测试：`test_install_helper_linux_has_no_trailing_backslash_n`

**经验**：
- raw f-string (`rf''`) 适合**没有**真换行需求的字符串（URL、JSON 片段、单行命令）
- 需要真换行时用普通 f-string (`f''`)
- 涉及"复制粘贴到终端"的内容，**末尾永远不要加任何换行字符**——终端本来就自带 Enter
- 在 raw f-string 里看到 `\n`、`\t`、`\r` 时**永远**要想：这是字面 2 字符，不是 escape

**本次修复文件**：
- `src/api/routers/agents.py` — `/install-helper` snippet 不带 trailing newline
- `frontend/src/pages/HostOnboardPage.tsx` — Windows fallback 改真换行
- `tests/unit/api/test_agents_api.py::TestInstall` — 回归测试

## 十、systemd heredoc 单引号 bug (2026-07-17, 用户反馈)

**用户反馈**：

```
[secagent] Created symlink /etc/systemd/system/multi-user.target.wants/secagent.service → /etc/systemd/system/secagent.service.
Failed to restart secagent.service: Unit secagent.service has a bad unit file setting.
```

**根因**：systemd 单元 heredoc 用了单引号包裹分隔符：

```bash
# 错的：
cat > /etc/systemd/system/${{SERVICE_NAME}}.service <<\'EOFSVC\'
# 改后：
cat > /etc/systemd/system/${{SERVICE_NAME}}.service <<EOFSVC
```

加了单引号的 heredoc (`<<\'EOF\'`) 会**禁用 bash 的变量展开**，所以 `$INSTALL_DIR` 字面写进了 `ExecStart=`，systemd 看到 `ExecStart=$INSTALL_DIR/agent` 直接拒绝。

**修复**：
- `src/agents/enroll.py:INSTALL_SH` 去掉 systemd heredoc 的单引号 → 改为 `<<EOFSVC`
- 保留 `<<\'EOFCFG\'` (config.json heredoc 用单引号，因为里面的 `$CONSOLE`/`$CONFIG_DIR` 是给 Go agent 运行时解析的字面占位符)
- `<<EOFBODY` (REG_BODY heredoc) 不用单引号，因为里面的 `$TOKEN`/`$HOSTNAME` 等需要在主机端就展开
- 加 1 条回归测试：`test_install_script_systemd_unit_expands_variables`

**bash heredoc 选用规则（已记入知识库）**：

| 用途 | heredoc opener | 原因 |
|---|---|---|
| 配置占位符（Go agent 运行时替换） | `<<\'EOF\'` | 防止 bash 提前展开 |
| shell 变量要在生成时定值（如 systemd 单元） | `<<EOF` | bash 展开 `$VAR` |
| 临时变量传递（如 JSON 请求体） | `<<EOF` | bash 展开 `$VAR` |

**经验**：每个 heredoc 都要问一句"这个文件交给下游时，shell 变量应不应该已经被解析？" 答错就要么部署后端报错、要么内容全是字面占位符。

## 十二、`_parse_ts`: ISO 字符串 → datetime (2026-07-17, 用户第二轮反馈)

**用户反馈**：上一轮修了 last_heartbeat 默认为 `""` 的问题后，立刻又 500：

```
asyncpg.exceptions.DataError: invalid input for query argument $11:
  '2026-07-17T12:56:22.756514+00:00'
  (expected a datetime.date or datetime.datetime instance, got 'str')
```

**根因**：上一轮把 `last_heartbeat` 设成了 `datetime.now(UTC).isoformat()` —— 有值了，但**仍然是字符串**。asyncpg 的 timestamptz 编码器只接受：
- `datetime.datetime`
- `datetime.date`
- `int`（unix timestamp）
- `None`

**不**接受任何形式的字符串。Pydantic 模型的 `str = ""` 默认值污染了数据流，必须在 store 层加一道解析。

**修复**：

```python
def _parse_ts(value):
    if value is None:
        return None
    if isinstance(value, datetime):
        return value        # passthrough
    if isinstance(value, str):
        s = value.strip()
        if not s:
            return None     # empty -> NULL
        if s.endswith("Z"):
            s = s[:-1] + "+00:00"   # Python <3.11 fromisoformat doesn't accept Z
        try:
            return datetime.fromisoformat(s)
        except ValueError:
            return None
    return None
```

应用点：
- `save_host` —— `last_hb = _parse_ts(getattr(host, "last_heartbeat", None))`
- `update_host` —— 处理 `last_heartbeat` 字段时空值 → `datetime.now(UTC)`，非空 → `_parse_ts(v) or now()`

**新增测试** `TestParseTs`（7 条）：覆盖 `None` / 空串 / 乱码 / datetime passthrough / ISO with offset / ISO with Z suffix / ISO with 不同 tz。

**关键经验**：

1. **Pydantic 字符串字段 + PG timestamptz = 必须显式 parse**。这是 LLM/Agent 写代码的常见盲点 —— `.isoformat()` 给的是字符串，asyncpg 不接受。
2. **不要让模型层污染 PG 类型**。所有写入 PG 时间戳的路径都要走 `_parse_ts`（或其它等价规范化函数）。
3. **同类问题可能蔓延**：项目里 `created_at`、`detected_at` 等字段在 Pydantic 模型里也是 `str = ""`。**还没在写入路径触发**，但保险起见：
   - 把 Host / ScanTask / ScanResult / VulnFinding 等模型里所有 `str` 类型的"应该是 datetime"的字段统一改成 `Optional[datetime] = None`
   - 或者在 store 层都过一遍 `_parse_ts`

## 十三、agent_id/agent_token 没写进 config.json → 启动崩溃 (2026-07-17, 用户反馈)

**用户反馈**：

```
$ systemctl status secagent
● secagent.service - Security Agent
   Active: activating (auto-restart) (Result: exit-code)
  Process: 856874 ExecStart=/opt/secagent/agent (code=exited, status=1/FAILURE)

$ journalctl -u secagent -n 50
agent[856874]: main.go:72: [agent] no agent_id configured and no enrollment token available
```

systemd 一直重启（`restart counter is at 97`），agent 一启动就退出。

**根因**：

`/etc/secagent/config.json` 内容（**安装时写的**）：

```json
{"console_url": "$CONSOLE", "ca_path": "$CONFIG_DIR/ca.pem"}
```

- 没有 `agent_id`
- 没有 `agent_token`
- `$CONSOLE` 是字面字符串（heredoc 用了 `<<'EOFCFG'` 单引号，bash 不展开）

Go agent 读到空 `AgentID` → `log.Fatal("no agent_id configured")` → 退出码 1 → systemd 重启循环。

**更深层的设计 bug**：原脚本的步骤顺序是

1. 下载 binary
2. **写 config.json（没有凭证）**
3. 写 systemd unit
4. **重启服务** ← 这时 config 还是空的！
5. POST /enroll
6. 拿返回但**没写回 config.json**

→ 服务是在没有凭证的情况下被启动的。

**修复（重排步骤）**：

| 原 | 新 |
|---|---|
| 写 config.json | 写 systemd unit（仅 enable，**不启动**）|
| 重启服务 | POST /enroll → 解析 JSON 拿 `agent_id`/`agent_token` |
| POST /enroll | 写 config.json（含真实凭证）|
| （拿到响应但不写回）| **重启服务**（仅当凭证齐全）|

**附带修复**：

- heredoc 改 `<<EOFCFG`（无引号）→ bash 展开 `$CONSOLE` / `$CONFIG_DIR` / `$AGENT_ID` / `$AGENT_TOKEN`
- 凭证提取：`python3 -c "import json,sys; ..."` 优先；无 python3 时 fallback 到 `sed`
- 重启条件化：`if [ -n "$AGENT_ID" ] && [ -n "$AGENT_TOKEN" ]` 才 systemctl restart，否则打 WARNING 让运维重跑
- 新增回归测试 `test_install_script_writes_agent_credentials_to_config`：检查步骤顺序、heredoc 类型、字段

**经验**：

> 1. **服务启动前所有配置必须就绪**。注册 → 拿到凭证 → 写 config → 启动。颠倒任何一步都会留下「服务跑在残缺配置上」的状态。
> 2. **涉及凭证的安装脚本**必须先成功获取凭证 → 写入 → 启动，**不能**先启动再写凭证。
> 3. **`<<'EOF'` 单引号 heredoc 在 bash 中禁用变量展开**，适合作为 JSON 模板（Go agent 运行时解析 `$VAR` 占位符）；但写入**实际配置**时必须 `<<EOF`（无引号）以展开当前 shell 变量。

**主机端验证步骤**：

```bash
# 1. 重跑安装脚本（会先写新 config 再启动 service）：
sudo bash secagent-install.sh

# 2. 检查 config.json 内容：
sudo cat /etc/secagent/config.json
# 应该包含 "agent_id": "agent-..." 和 "agent_token": "..."

# 3. 检查服务状态：
systemctl status secagent
# 应该是 active (running)，不再 activating (auto-restart)
```

## 十四、INSTALL_SH 末尾残留片段 → bash syntax error (2026-07-17, 用户反馈)

**用户反馈**：

```
[secagent] Installation complete. Service: secagent (status: active)
secagent-install.sh: line 143: syntax error near unexpected token `('
```

（实际服务已经起来了，但脚本最后一行失败，operator 看到错误提示以为没装好。）

**根因**：上一轮重排 install 步骤时，旧的 `log "Installation complete..."` 行被替换成新的，但**只替换了 `log ` 那一半**，留下了尾巴：

```
log "Installation complete. Service: ${SERVICE_NAME} (status: $(systemctl is-active ${SERVICE_NAME} 2>/dev/null || echo unknown))"
Service: ${SERVICE_NAME} (status: $(systemctl is-active ${SERVICE_NAME} 2>/dev/null || echo unknown))"
"""
```

第二行没有 `log ` 前缀，bash 把它当成命令执行 → `Service:` 是命令名、`(` 后面被当成函数调用语法 → `syntax error near unexpected token`。

**修复**：用 bytes 级别正则定位并删除残留的 `Service: ${SERVICE_NAME} ...))"

` 行。

**新增回归测试** `test_install_script_has_no_garbage_after_last_log`：渲染 install 脚本后，找到 `Installation complete.` 标记，取它之后的全部内容（应该是末尾的 `
` 或空白），assert `strip()` 之后为空字符串。任何下次类似的残留会立即被该测试捕获。

**经验**：

> 1. **替换字符串时，要么完整匹配整块，要么用 regex + 多行匹配**。不要用 `replace("log ...", "log ...") ` 这种只覆盖一半的替换 —— 留下另一半就成了悬空的 bash。
> 2. **多行字符串（`INSTALL_SH = """..."""`）的编辑**：每次改动后**强制 `import ast; ast.parse(open(...).read())`**，并且把渲染后的脚本和上一次 `diff` 对比，发现多出来的行立即回滚。
> 3. **回归测试要测「前后结构」而不只是「期望行存在」**。本轮的测试不只检查"必须有 log 'Installation complete'"，还检查"它后面不能有任何非空白字符"，这样能捕获所有「末尾残留」类型的 bug。
> 4. **多轮反复 edit Python 源里的多行字符串是高风险动作**。建议下次写到专门文件（`install.sh.tmpl`）由 Python `read()` 加载 + `format()` 渲染，比把整段 shell 嵌进 Python 字符串要可控得多。

## 十五、HTTP_CODE="200000" 拼接 + curl code 23 误判 (2026-07-17, 用户反馈)

**用户反馈**：

```
[secagent] Downloading agent binary from http://192.168.254.121:8000 ...
curl: (23) Failure writing output to destination
secagent-install.sh: line 46: warning: command substitution: ignored null byte in input
[secagent][ERROR] Agent binary download failed.
  URL        : http://192.168.254.121:8000/api/v1/agents/binary/linux/amd64
  HTTP code  : 200000
  Server said:ELF>@G@@8@@@@@@PPxx@x@dd@@mLmLpLpp...
```

（特别注意：`HTTP code  : 200000` — 多出来的 "000" 是 fallback 拼接出来的。）

**根因**（双重 bug）：

```bash
HTTP_CODE=$(curl -sS -o "$INSTALL_DIR/agent" -w "%{http_code}" \
    -H "Authorization: Bearer $TOKEN" "$BIN_URL" || echo "000")
```

1. **`$()` 捕获整个子 shell 的 stdout**。当 curl 退出非零（即使 body 已经写完），它**先**把 `-w "%{http_code}"` 的 "200" 打印到 stdout，**然后**失败
2. `|| echo "000"` 触发，**也**打印到 stdout
3. `$()` 把 "200" + "000" 拼成 "200000"
4. `if [ "200000" != "200" ]` 误判失败 → 进入错误分支
5. `head -c 400 "$INSTALL_DIR/agent"` 显示**已下载好的 ELF 二进制**当错误信息

至于 `curl: (23)`：是 curl 自己的副作用（在收到 response body 后写文件时偶尔失败，例如服务器关闭了 keep-alive 连接）。**文件实际已下完**，只是 curl 退出码非零。

**修复（双重防御）**：

```bash
# 1. 把 || 拆出来，避免拼接
HTTP_CODE=$(curl -sS -o "$INSTALL_DIR/agent" -w "%{http_code}" \
    -H "Authorization: Bearer $TOKEN" "$BIN_URL" 2>/dev/null) || true
# 2. 仅当 HTTP_CODE 完全为空时才 fallback 到 "000"（参数展开）
HTTP_CODE="${HTTP_CODE:-000}"

# 3. 即使 HTTP_CODE 不是 200，但文件是合法 ELF，也算成功
IS_ELF=0
if [ -s "$INSTALL_DIR/agent" ] && head -c 4 "$INSTALL_DIR/agent" 2>/dev/null | grep -aq "ELF"; then
    IS_ELF=1
fi
if [ "$HTTP_CODE" != "200" ] && [ "$IS_ELF" -ne 1 ]; then
    # ...真正的失败分支
fi
```

**新增回归测试** `test_install_script_handles_curl_partial_write`：断言
- `|| echo "000")` 这种拼接模式已不存在
- 必须使用 `HTTP_CODE="${HTTP_CODE:-000}"`
- 必须有 `grep -aq "ELF"` 校验文件合法性

**经验**：

> 1. **`$()` 捕获整个子 shell 的 stdout**。`$(cmd || echo "000")` 在 cmd 失败时会**拼接** cmd 的 stdout + "000"，因为 `||` 是 shell 语法不是字面替换。
> 2. **fallback 不要和正常 stdout 共享同一个 `$()`**。模式 `$(cmd) || fallback` 比 `$(cmd || fallback)` 更安全。
> 3. **校验完整性优于校验 exit code**：网络/磁盘异常经常让 curl/wget 退出非零但 body 完整。**额外校验文件本身**（magic bytes、大小、签名）比单纯信任 exit code 更可靠。
> 4. **设计原则：success path 不应依赖单一信号**。状态码 + 文件头校验 + 大小检查 三选二才判成功，比单纯 `code==200` 更稳健。

## 十六、主机纳管 UX 收尾：同 IP 去重 + rule_version 初始同步 (2026-07-17, 用户反馈)

**用户反馈**：

> 我用三个不同的令牌在同一台主机执行脚本后，控制台主机列表会有三个记录。我觉得主机 IP 应该保持唯一性，根据 IP 保留最新的那条记录。还有规则版本和组为什么是空的，需要什么条件？

**问题诊断**：

| 项 | 现象 | 根因 |
|---|---|---|
| 同 IP 三条记录 | 每次 enroll 创建新 agent_id，PG 里多行 | api_enroll_host 从来不查重 |
| 规则版本空 | UI 显示 - | /enroll 响应不带 rule_version；agent 启动时为空，要等服务端推 rule_update 命令 |
| 组空 | UI 显示 - | 创建 token 时没选组（前端组下拉 allowClear）。**这是 by design**——选择可选，不选就空 |

**修复**：

### 1. 同 IP 去重（保留最新）

src/agents/manager.py 新增 decommission_host_by_ip()。api_enroll_host 在创建新行之前，先把同 IP 的旧行 decommission 掉（连同 Redis 在线标记）。

### 2. rule_version 初始同步

EnrollResponse 模型新增字段 rule_version。api_enroll_host 调用 current_rule_version() 写到响应。INSTALL_SH 抓 rule_version 字段（python3 优先，sed fallback），写入 config.json：

```bash
cat > "$CONFIG_DIR/config.json" <<EOFCFG
{"console_url": "$CONSOLE", "ca_path": "$CONFIG_DIR/ca.pem",
 "agent_id": "$AGENT_ID", "agent_token": "$AGENT_TOKEN",
 "rule_version": "$RULE_VERSION"}
EOFCFG
```

主机 UI 立即显示真实的规则版本，不再是 -。

### 3. 组（group）

保持 by design——允许空组。下次创建 token 时可选填 生产/测试/开发，enroll 后 host 表会带上组名。

**经验**：

> 1. 同一物理实体的多次 enroll 应该有去重逻辑。最朴素的：用 IP 做主键，后入覆盖前入。需要 NAT/proxy 提示，但比留一堆幽灵主机好。
> 2. enroll 响应应当自包含——客户端拿到响应就能开干，不应该再调一次 GET /latest_rule_version。否则 enroll token 又要带另一组权限。
> 3. 避免空字段污染 UI。任何 Pydantic 字符串字段默认值是空串又会写进强类型列（如 timestamptz），都会埋雷。规则：默认值用 Optional[X] = None，store 层做规整。
> 4. 测试也要避免脆弱字面量。这次新加的 2 条测试里我用单引号 Python 字符串里嵌套单引号——Python 解析直接报 unterminated literal。下次用 chr(34) 拼接或 raw string 兜底。

新增 2 条测试：
- test_install_script_extracts_rule_version_from_enroll
- test_enroll_response_includes_rule_version_field
