# 后续计划 V5（2026-07-22）

> 基于《项目审查V5》的剩余问题，按功能影响优先级排列

## 第一优先级：P1 严重（影响核心页面可用性）

### 1. DashboardPage 图表崩溃修复
**问题**: 首页运营大屏用 @ant-design/charts（Pie/Column/Line），数据为空或库加载失败时白屏崩溃。ScanReportPage 已因同样问题改成 antd Progress。
**方案**:
- DashboardPage 的 3 个图表（结论分布 Pie / 定级分布 Column / 趋势 Line）改用 antd 原生组件
  - Pie/Column -> Progress 条形图或 Statistic 卡片
  - Line -> 简单的 Table 或 Progress 趋势条
- 或加 ErrorBoundary 包裹图表区域（兜底，但不如替换彻底）
**工作量**: 1-2 小时
**文件**: `frontend/src/pages/DashboardPage.tsx`

### 2. EventQueuePage/EventDetailPage SSE cleanup 统一 ref 模式
**问题**: 这两个页面的 EventSource cleanup 用闭包变量，有极小竞态窗口。ScanMonitorPage 已用 eventSourceRef 修复。
**方案**:
- EventQueuePage: 加 esRef（useRef），cleanup 关 ref
- EventDetailPage: 同上
- 统一 4 个 SSE 页面的 cleanup 模式
**工作量**: 30 分钟
**文件**: `frontend/src/pages/EventQueuePage.tsx`, `frontend/src/pages/EventDetailPage.tsx`

## 第二优先级：P2 一般（体验/健壮性）

### 3. vite proxy SSE 缓冲优化
**问题**: dev 模式 vite proxy 对 SSE 有缓冲，实时事件略有延迟。
**方案**: vite.config.ts proxy 加 SSE 透传配置，或文档说明生产 nginx 配 `proxy_buffering off`。
**工作量**: 15 分钟
**文件**: `frontend/vite.config.ts`

### 4. ScanMonitorPage findings 重复拉取
**问题**: refreshTask 在 completed/failed 时拉 findings，防抖刷新可能重复拉。
**方案**: findings 加 state 标记，已拉过不重复拉。
**工作量**: 15 分钟
**文件**: `frontend/src/pages/ScanMonitorPage.tsx`

### 5. 主机组删除后 hosts.group_name 残留
**问题**: 删组后主机保留旧组名，list_groups 显示 legacy 孤儿。
**方案**: delete_group 时可选清空 hosts.group_name，或前端提示"组内主机将变为未分组"。
**工作量**: 30 分钟
**文件**: `src/agents/store.py`, `src/api/routers/agents.py`

### 6. 删除任务清 Redis 缓存
**问题**: 删任务后 GET 仍可能返回 Redis `queue:status` 缓存。
**方案**: delete_task 时 `redis.delete(f"vulnscan:queue:status:{task_id}")`。
**工作量**: 10 分钟
**文件**: `src/agents/store.py` 或 `src/api/routers/vulnscan.py`

### 7. 对话式扫描 intent 解析容错
**问题**: LLM 解析 intent 不准时 targets/modules 可能错误。
**方案**: IntentCard 已允许用户确认/修改，当前可接受。后续可加 intent 校验（targets 必须是已纳管主机/组）。
**工作量**: 1 小时（后续）
