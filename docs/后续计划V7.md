# 后续修复计划 V7（基于项目审查 V7）

> 目标：先恢复升级、取消等控制命令的真实端到端语义，再恢复可重复的发布门禁，最后补齐 Nuclei、缓存和配置中心等部分实现。

## 一、执行原则

1. 每个接口返回成功前，必须能证明命令已被目标组件接受，不能用“已写状态”代替“已执行动作”。
2. P0/P1 修复必须带回归测试；涉及跨进程或外部资源的链路必须有集成测试。
3. 不继续在 `ChatScan.tsx` 内叠加职责，先收拢消息持久化和任务创建边界。
4. Nacos 热更新按配置所有权分级：可动态更新的字段即时生效；连接池类字段明确重建或标记“需重启”。

## 二、阶段 0：P0 控制链路（阻断发布）

### 0.1 打通扫描取消的端到端链路

**涉及文件**：

- `src/api/routers/vulnscan.py`
- `src/agents/ws_gateway.py`
- `src/agents/store.py`
- `agent/internal/scan/engine.go`
- `agent/internal/scan/collector.go`
- `agent/internal/scan/nuclei/runner.go`

**实施步骤**：

1. 为任务保存实际下发的 `agent_id` 集合，取消接口向这些 Agent 发送签名 `scan_cancel`。
2. 增加 `cancelling`/`cancelled` 状态，不再把取消伪装成 `failed`。
3. 将 `Execute` 根 context 传给 matcher 采集、匹配和 `runNuclei`，删除 Nuclei 内部独立的 `context.Background()`。
4. 取消后发送明确的 cancel ack/final event；服务端以幂等方式收敛到 `cancelled`，忽略晚到结果或标记为 late result。
5. 对“任务已完成”“Agent 离线”“重复取消”“跨 worker Agent”定义稳定返回值。

**验收标准**：

- API 返回成功后，运行中的 matcher 与 Nuclei 进程都在限定时间内退出。
- 任务最终状态为 `cancelled`，不会被晚到结果改回 completed。
- Python 路由/网关测试、Go context 测试和一条真实 Agent 集成测试通过。

**估时**：4-6 小时。

### 0.2 重做 Agent 升级为两阶段、可回滚流程

**涉及文件**：

- `src/api/routers/agents.py`
- `src/agents/signing.py`
- `src/agents/ws_gateway.py`
- `agent/cmd/agent/main.go`
- `agent/internal/updater/upgrade.go`
- `agent/internal/updater/upgrade_test.go`

**实施步骤**：

1. 服务端根据目标 Agent 的 OS/arch 选择受控二进制，不再依赖客户端提交任意 `download_url`。
2. 对二进制 SHA-256 签名，将 `signature` 放入升级 payload；保留外层 WS 命令签名，两种签名职责分离。
3. 下载端点支持 `agent_id + Authorization: Bearer <agent_token>`，并与 enrollment token 下载路径分开。
4. `HandleUpgrade` 检查 `NewRequest`、下载、chmod、两个 rename 的每一个错误；任何失败都回滚旧文件。
5. 移除 updater 内部直接 `os.Exit`。先发送“staged/verified”确认，由 systemd/Windows service helper 完成原子替换和重启；新进程首次心跳上报目标版本作为最终成功依据。
6. 为文件替换和退出函数做依赖注入，覆盖成功、签名错误、第二次 rename 失败、回滚失败等测试。

**验收标准**：

- 内置升级 API 可完成真实 Agent 升级，并在控制台看到 staged、restarting、confirmed 三阶段状态。
- 任一步失败都保留可启动的旧二进制。
- Agent token 不能访问 enrollment-only 资源，enrollment token 也不能冒充已注册 Agent。

**估时**：6-10 小时。

## 三、阶段 1：P1 正确性与发布门禁

### 1.1 修复主机物理删除参数

**文件**：`frontend/src/api/client.ts`、`frontend/src/pages/HostOnboardPage.tsx`。

- 将 `_force` 改为 `purge`，请求参数使用 `{ params: { purge } }`。
- 增加前端 API 单测或 Playwright 路由断言，确认第二次删除发出 `DELETE ...?purge=true`。

**验收**：已下线主机物理删除后，PG、ES、Redis 与列表均无残留。

**估时**：0.5-1 小时。

### 1.2 统一聊天响应与持久化

**文件**：`src/api/routers/chat.py`、`src/api/routers/scan_chat.py`、`src/agents/conversation.py`、`frontend/src/api/client.ts`、`frontend/src/components/ChatScan.tsx`。

- 让统一 `/chat` 在同一次调用中持久化“原始用户消息 + 实际返回给用户的 reply + route/sources/intent”。
- 删除 `chatAssistant()` 内第二次 `chatConversation()` 调用。
- 执行扫描时只记录结构化 action/message，不再把全部历史拼成新用户消息并再次调用模型。
- 用户消息和助手消息使用一次事务追加，避免半条对话。
- 将 conversation sidebar、message flow、intent card 拆成独立组件/模块。

**验收**：每发送一条消息只发生一次回答模型调用；刷新前后消息文本一致；执行扫描不新增重复问答。

**估时**：4-6 小时。

### 1.3 让 `config_update` 真正生效并修复自保护

**文件**：`agent/cmd/agent/main.go`、`agent/internal/comm/client.go`、`agent/internal/protection/protection.go`。

- 给 heartbeat loop 增加可重置 ticker 或动态 interval channel。
- 将 resource limit 映射到唯一的 protection 模块，删除/收敛旧 `resource.Monitor` 双实现。
- 修正 `SampleInterval=5s` 与 `CPUPollTimeout=1s` 的矛盾；不同探针使用独立 context，错误可观测。
- 应用 log level；需要跨重启保留的配置原子写回 `config.json`。
- 只有全部字段应用成功后才发送成功 ack，部分失败返回字段级错误。

**验收**：在线修改心跳后，下一周期按新间隔发送；构造高 CPU/内存探针时扫描被暂停；重启后持久字段保留。

**估时**：4-6 小时。

### 1.4 完成主机下线与凭证撤销语义

**文件**：`src/agents/manager.py`、`src/agents/enroll.py`、`src/agents/ws_gateway.py`、`src/api/routers/agents.py`。

- 在同一事务内将 host 标记 decommissioned 并设置 `agent_tokens.revoked_at`。
- 关闭本 worker 的 WebSocket；跨 worker 通过 Redis control channel 请求连接所有者断开。
- heartbeat/register_online 拒绝 revoked/decommissioned Agent。
- 如需恢复，提供显式 re-activate/re-enroll 流程，不允许心跳自动复活。

**验收**：下线后连接立即关闭，原 token 无法重连，主机状态不会被后台心跳改回 online。

**估时**：3-4 小时。

### 1.5 修复跨 worker 投递结果

**文件**：`src/agents/ws_gateway.py` 及调用方。

- 检查 `redis.publish()` 返回的 subscriber 数；大于 0 返回 accepted，而不是固定 False。
- 更可靠的方案是加入 command id 与 Agent ack，API 返回 `accepted`，状态查询返回最终 ack。
- 对重试做幂等去重，避免重复扫描/升级。

**验收**：双 uvicorn worker 下，Agent 连接在 worker A、请求落到 worker B 时不再误报 404，命令只执行一次。

**估时**：2-4 小时。

### 1.6 修正 Nacos 启动顺序和热更新边界

**文件**：`src/api/main.py`、`src/common/config/settings.py`、`src/common/config/nacos_loader.py`、各连接池模块。

- 在创建 ES/PG/Redis 客户端前完成首次 Nacos 加载。
- 无论首次 fetch 是否成功都启动带 backoff+jitter 的 poller，使 Nacos 恢复后自动加载。
- 记录由 Nacos 注入的 key 集合；配置删除时清除旧注入值，但绝不覆盖进程启动时的显式 env。
- 将字段分成“动态生效”和“需重建/重启”；对 PG/ES/Redis 连接配置实现受控重建，或明确拒绝伪热更新。
- 防止重复 listener，并在 shutdown 中 await cancellation。

**验收**：仅设置 Nacos 连接参数即可冷启动；启动时 Nacos 离线、随后恢复时能自动加载；删除配置键后不会继续使用旧值。

**估时**：5-8 小时。
