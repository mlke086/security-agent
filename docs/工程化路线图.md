## 二、七项任务：思路与开发步骤

### 任务 1：审批队列外置 Redis，支持多 worker / 多进程

**现状定位**：`hitl_handler.py:28` 的 `_pending_approvals: dict` 是模块级内存字典；`hitl_approval_node`（:142）用 `asyncio.sleep(5)` 轮询它；`operations.py:103` 的 `approve_event` 通过 `resolve_approval_by_event_id` 写同一个字典。**只有单进程内才闭环**——多 worker（uvicorn `--workers N`）或独立消费进程下，API 在 worker A 审批、HITL 在 worker B 轮询，永远等不到。

**思路**：把审批状态从"进程内字典"迁到"Redis 共享存储"，用 Lua 脚本保证多人 quorum 的原子性。中期目标是用 LangGraph 的 `interrupt()/checkpointer` 彻底替代轮询。

**Redis 数据结构设计**
```
approval:{approval_id}        Hash  status/required/event_id/level/created_at
approval:{approval_id}:voters Set   已审批人（去重，支持 quorum）
approval:pending              ZSet  score=created_at ts（超时扫描+列表）
approval:event:{event_id}     Set   反查（event->approval_id）
approval:notify:{approval_id} Pub/Sub  决策事件（供 HITL 等待/任务5 SSE）
```

**开发步骤**
1. 新建 `src/orchestration/subgraphs/responder/approval_store.py`，定义 `ApprovalStore`（基于 `redis.asyncio`）：`create / get / list_pending / add_vote / resolve / wait_result`。
2. `add_vote` 用 Lua 脚本原子执行：`SADD voters → SCARD → 若 ≥required 则 HSET status=approved`，返回最终状态，避免并发超批。
3. `wait_result(approval_id, timeout)`：先查 Hash，未决则 `SUBSCRIBE approval:notify:{id}`，超时则置 `timeout`（替换现在的 `while+sleep`）。
4. 改造 `hitl_handler.py`：`hitl_approval_node` 调 `ApprovalStore.create`+`wait_result`；删除 `_pending_approvals`、`_add_pending_approval`、`resolve_approval_*` 的内存实现（保留同名函数转调 store，避免大改调用方）。
5. 改造 `operations.py`：`approve_event` 调 `ApprovalStore.add_vote(actor, decision)` + `PUBLISH` 通知；`get_approvals` 调 `list_pending`。
6. 修 `celery_app.py:19` 的 `approval_timeout_task`：把不存在的 `resolve_approval` 改为 `ApprovalStore.resolve(approval_id, "timeout")`，作为超时兜底（双保险，即使 HITL 进程崩了也能超时关单）。
7. **多 worker 验证**：`uvicorn ... --workers 3`，用 worker A 的 API 审批、确认 worker B 上的 HITL 能收到并继续 execute。

**验收**：3 worker 下 L4 双人审批跨进程达成 quorum；杀掉持有 HITL 的进程后，Celery 超时任务仍能把单据置为 timeout。

> 中期重构：引入 LangGraph `RedisSaver` checkpointer + `interrupt()`，审批后用 `Command(resume=...)` 恢复图执行，彻底去掉轮询。可作为任务 1 的 v2。

### 任务 2：事件持久化到 ES/DB（当前内存仓，重启即失）

**现状定位**：`api/store.py:42` `EventStore` 是 `dict[str, EventRecord]` 单例，进程重启即空，多 worker 间也不共享。`settings.py:48` 已预留 `es_index_events="security-agent-events"` 但从未使用。`/trace` 端点（`operations.py:45`）已尝试 fallback 查 ES audit，但 trace 步骤本身只存在内存 `ev.trace`。

**思路**：用 ES 同时作为**事件主存储**和**轨迹存储**（统一索引、按 `event_id` 聚合），保留 `EventStore` 接口名、替换底层实现。一举解决：持久化、多 worker 共享、trace 检索三件事，并与"后续计划V2 修复6（全节点审计落 ES）"合流。

> 选型（结合 0.2 实测）：ES 已集成、`es_index_events` 已预留，作主存最省事；同机房另有 **MySQL**（事务型，适合事件状态机多次原子更新）与 **ClickHouse**（聚合极快，适合指标/趋势读模型）。首版用 ES 落地；量大时指标/趋势读模型再迁 ClickHouse，事件状态机若变复杂可引入 MySQL 做事务主存、ES 做检索副本。

**索引设计**
```
security-agent-events   _id=event_id   摘要字段（status/verdict/priority/duration…）
security-agent-audit    _id=uuid        每个节点一步一个 doc，含 event_id+node+action+details+timestamp
                         /trace 端点按 event_id 聚合 audit 即得推理链
```

**开发步骤**
1. 新建 `src/api/store_es.py`，实现 `ESEventStore`（与 `EventStore` 同方法签名）：
   - `create_event` → `index(es_index_events, id=event_id, doc=...)`
   - `update_event` → `update` + 乐观并发（`if_seq_no/if_primary_term` 或 `version_type=external`），冲突重试 2 次。
   - `add_trace_step` → 写一条 audit doc（复用 `AuditLogger`），不再维护内存 list。
   - `list_events` → ES `search` + filter + sort by submitted_at + 分页。
   - `get_event` → `get(es_index_events, id=event_id)`，trace 用 audit 聚合补全。
   - `metrics` → ES `terms` 聚合（by_verdict/by_priority）+ `avg` 聚合（duration）。
2. 在 `get_event_store()` 增加 `backend` 开关（`store_backend: Literal["memory","es"]="es"`），方便回退和测试。
3. **全节点审计落 ES**：`entry/orchestrator/aggregator` 各加一行 `await audit_node(...)`（后续计划V2 修复6 的 helper），保证 trace 完整。
4. 启动时建索引 mapping（事件字段 + audit 的 `event_id` keyword 用于聚合/term 查询）。
5. `_run_pipeline`（`main.py:55`）里写内存 trace 的逻辑（:71-78）改为依赖 store 自动落 ES，去掉手工 `add_trace_step` 循环。

**验收**：提交事件→重启 API→`GET /api/v1/events/{id}` 仍在、trace 完整；`--workers 3` 下三个 worker 看到同一份数据；`/metrics` 来自 ES 聚合。

### 任务 3：`execute_response` 接入真实处置动作

**现状定位**：`hitl_handler.py:158` `execute_response_node` 只是 `for op in operations: executed.append({"type":..., "status":"executed"})`——**纯空转**，没有任何真实调用。`playbook_matcher.py` 已能产出 `Operation(type, level, params)`，10 个 YAML 剧本就绪。

**思路**：建一个**动作连接器（Action Connector）层**，按 `op.type` 路由到具体适配器，统一接口、统一审计、统一安全闸。关键原则：**默认 dry-run**、**审批后才执行**（已由 HITL 保证）、**幂等+可回滚+超时**。

**接口设计**
```python
class ActionConnector(Protocol):
    op_types: list[str]
    async def execute(self, op: Operation, ctx: ActionContext) -> ActionResult: ...
    async def rollback(self, op: Operation, ctx: ActionContext) -> None: ...   # best-effort

class ActionDispatcher:
    def __init__(self, dry_run: bool = True): self._registry: dict[str, ActionConnector] = {}
    async def execute_playbook(self, playbook, event_id) -> list[ActionResult]:
        # 逐 op 调用，每步落 ES audit，失败则回滚已执行步
```

**连接器清单（按实验室可用性分层实现）**

| op.type | 适配目标 | 实验室可行性 |
|---|---|---|
| `notify_analyst` | 复用 `notifier.py` 企微/钉钉 webhook | ✅ 立即可做 |
| `siem_tag` | ES `update` 给事件打标签 | ✅ |
| `dns_block` | 写 Redis blocklist `dns:blocklist` set | ✅ |
| `firewall_block` | 调靶机容器 iptables / 或 FortiGate API | ⚠️ 需靶机 |
| `isolate_host` / `terminate_process` / `quarantine_file` | EDR API（CrowdStrike/奇安信） | ⚠️ 需 EDR |
| `edr_policy` | EDR API | ⚠️ 需 EDR |

**开发步骤**
1. 新建 `src/execution/actions/` 包：`base.py`（Protocol/ActionResult/ActionContext）、`dispatcher.py`、`connectors/{notify,siem_tag,dns_block,firewall,edr}.py`、`simulator.py`（dry-run 默认实现，记录意图到 ES）。
2. `ActionResult`：`status(success|failed|skipped|dry_run)/output/error/op_id/started_at/finished_at`。
3. `ActionContext` 携带 `event_id/approval_id/actor/dry_run`，每步执行前后写 ES audit（`action=execute_op`/`execute_op_rollback`）。
4. 改造 `execute_response_node`：注入 `ActionDispatcher`，调 `execute_playbook`，把 `executed_operations` 换成真实 `ActionResult` 列表。
5. 加全局开关 `action_dry_run: bool = True`（settings），生产切 false 前必须配好真实连接器；L4/L5 动作额外要求 `dry_run=false` 时强制双人审批（已具备）。
6. 幂等：`op_id = hash(event_id+op.type+params)`，Redis `SETNX action:done:{op_id}` 防重复执行。
7. 先实装 `notify_analyst`/`siem_tag`/`dns_block` 三个真实可跑的，其余留 `simulator` 占位并 `log.warning("connector_not_implemented")`。

**验收**：构造 true_positive/conf≥0.8 事件→命中剧本→审批通过→`notify_analyst` 真实推送、`siem_tag` 真实落 ES、`dns_block` 真实写 Redis；`dry_run=true` 时只审计不执行；重复执行同 op_id 被跳过。

### 任务 4：`consumer._emit` 接入主图，打通 Kafka 实时告警入口

**现状定位**：`consumer.py:84` `_emit` 只 `logger.info("event_processed")`。主图只在 `main.py:55` `_run_pipeline`（API BackgroundTask）里被调用。**Kafka 入口和主图完全断开**，且 `_run_pipeline` 与 API 强耦合。

**思路**：抽取共享 `run_pipeline()`，让 API 和 Consumer 共用；Consumer 侧加**有界并发池**防止 LLM 重调用压垮。

**开发步骤**
1. 新建 `src/orchestration/runner.py`：
   ```python
   async def run_pipeline(event_id, text, iocs, source):
       store = get_event_store()
       store.create_event(event_id, text, iocs, source)   # 任务2后落 ES
       graph = get_compiled_graph()
       result = await graph.ainvoke({"event_id":event_id, "raw_event":{...}, "audit_log":[]})
       # 写回 store/trace/verdict/pending_approval（复用现 main.py 逻辑）
   ```
2. `main.py:_run_pipeline` 改为薄封装转调 `runner.run_pipeline`（保持 BackgroundTask 行为不变）。
3. `consumer._emit` 改为：
   ```python
   async with _sem:  # 模块级 asyncio.Semaphore(pipeline_concurrency)
       await run_pipeline(event["event_id"], event["sanitized_text"], event["iocs"], "kafka")
   ```
4. **并发控制**：模块级 `asyncio.Semaphore(settings.pipeline_concurrency=4)` 包住 `run_pipeline`；满载时新消息在 Kafka 层 backlog（已 `enable_auto_commit=False`，处理完才 commit，天然反压）。
5. **可选增强**：把 `run_pipeline` 包成 Celery task（已有 `celery_app.py` + Redis broker），Consumer 只做轻量 sanitize/extract 后投递 task，真正水平扩展靠 Celery worker 池。这样 Consumer 进程不阻塞在 LLM 上。
6. 错误处理：`run` 循环里 `_emit` 抛错走 DLQ（已有 `_send_dlq`），但要把"图谱执行失败"与"消息解析失败"区分——前者可重试队列、后者进 DLQ。
7. 启动入口：提供 `python -m src.preprocessing.consumer`（已有 `run_consumer`），与 API 解耦部署。

**验收**：往 `raw-alerts` topic 投一条蜜罐告警→Consumer 自动 sanitize/extract→进主图→ES 出现该 event_id 的完整 trace+verdict；前端事件队列能看到 `source=kafka` 的事件；并发投 20 条不 OOM（信号量限流）。

### 任务 5：SSE/WebSocket 替代轮询，实时推送轨迹

**现状定位**：`DashboardPage.tsx:19` 每 10s 轮询 `/metrics`；`EventDetailPage.tsx:27` 只在挂载时取一次，**推理链不实时更新**；`EventQueuePage` 大概率也轮询。无任何推送通道。

**思路**：用 **Redis Pub/Sub 作为跨 worker 事件总线** + **FastAPI SSE** 下发。SSE 单向推送足够（WebSocket 在此场景无额外收益，且 SSE 走 HTTP 更省事、更易过鉴权）。这条总线天然依赖任务 1（审批变更发布）和任务 2（事件/trace 变更发布）。

**通道设计**
```
Redis Pub/Sub:
  events:{event_id}   事件状态/trace 步变更
  metrics             指标更新（事件完成时发布）
  approvals           审批新建/决策
API SSE 端点:
  GET /api/v1/events/{id}/stream   订阅 events:{id}+approvals
  GET /api/v1/metrics/stream       订阅 metrics
  GET /api/v1/events/stream        订阅全局事件流（队列页）
```

**开发步骤**
1. 新建 `src/api/events_bus.py`：`EventBus` 封装 `redis.asyncio` pub/sub，`publish(channel, payload)` / `async subscribe(channel) -> AsyncIterator`。
2. 在任务 2 的 `ESEventStore.update_event/add_trace_step`、任务 1 的 `ApprovalStore.resolve`、`_run_pipeline` 完成处，调用 `bus.publish(...)`。**发布点即状态变更点**。
3. 新增 SSE 端点（`sse-starlette` 或手写 `StreamingResponse`+`text/event-stream`）：每个连接订阅对应 Redis channel，转发给客户端；带 JWT 鉴权（复用 `require_role`）；心跳 every 15s 防代理超时。
4. 前端 `api/client.ts` 加 `streamEvent(eventId, onMessage)` 用 `EventSource`；`EventDetailPage` 改为挂载取首屏 + SSE 增量追加 trace 步骤、实时刷新 status/approval；`DashboardPage` 改 SSE 订阅 metrics，去掉 `setInterval`。
5. 断线重连：`EventSource` 自带重连，记录 `Last-Event-ID` 做断点续传（可选，用 Redis stream 替代 pub/sub 即可获得回放能力——若需要"补漏"建议直接用 Redis Stream）。
6. 多 worker：任一 worker 发布，所有 worker 的 SSE 连接都能收到（pub/sub 天然 fan-out）。

**验收**：提交事件后，EventDetail 页面推理链逐步实时出现（不用刷新）；审批后状态秒级变绿；Dashboard 数字随事件完成实时跳动；3 worker 下推送不丢。

> 进阶：若要保证"离线期间产生的变更不丢"，把 pub/sub 升级为 Redis Stream（`XADD`/`XREAD`），SSE 端用 `last-event-id` 续读。

### 任务 6：前端 API 响应强类型、错误边界、E2E 测试

**现状定位**：
- `types.ts` 松散（`status: string`、`approvals: any[]`、`details: Record<string,any>`）；`client.ts` 靠 `as` 断言，后端 `operations.py` 返回裸 dict、**无 `response_model`**，OpenAPI schema 不权威。
- `App.tsx` 无 ErrorBoundary，组件抛错白屏。
- 无任何 E2E。

**思路**：后端用 Pydantic `response_model` 收口 → 前端从 OpenAPI 生成类型 → 加 ErrorBoundary + 统一错误拦截 → Playwright E2E。

**开发步骤**
1. **后端响应模型化**：在 `operations.py`/`main.py` 给每个端点加 `response_model`：
   ```python
   class EventListResponse(BaseModel): items: list[EventResponse]; total: int
   class TraceResponse(BaseModel): event_id: str; trace: list[TraceStep]; ...
   ```
   复用/收紧 `store.py` 的 `EventRecord`/`TraceStep`，`status` 改 `Literal["processing","completed","pending_approval","ignored","error"]`。
2. **前端类型生成**：`npm i -D openapi-typescript`，脚本 `openapi-typescript http://localhost:8000/openapi.json -o src/api/schema.ts`；用 `openapi-fetch` 替换 axios 得到端到端类型推断（或保留 axios 但 `as` 改为引用 schema 生成的类型）。删除手写 `types.ts` 或仅留纯前端类型。
3. **错误边界**：新建 `components/ErrorBoundary.tsx`（class component，`componentDidCatch` 上报），在 `App.tsx` 包裹 `<Routes>`，展示 antd `Result` 降级 UI + 重试按钮。
4. **统一错误拦截**：扩展现有 axios interceptor（`client.ts:20`）——按 HTTP 状态/业务码映射 antd `message`/`notification`，401 之外的网络错误/5xx 给统一提示；加请求超时与重试。
5. **运行时校验（可选）**：在 API 边界用 `zod` 校验关键响应（防止后端 schema 漂移），失败走 ErrorBoundary。
6. **E2E（Playwright）**：`npm i -D @playwright/test`，5 个场景：① 登录 ② 提交事件→队列出现 ③ 详情页推理链渲染 ④ 审批→状态变更 ⑤ 大屏指标。配 `playwright.config.ts` 指向运行中的前后端；CI 加 job。
7. 配合任务 5：E2E 里可断言 SSE 推送的 trace 步骤数。

**验收**：`tsc --noEmit` 零错误；故意传错字段后端 422；拔掉后端前端不白屏而是降级 UI；`playwright test` 5 场景全绿并进 CI。

### 任务 7：Dashboard 引入图表库（当前手写 div 条形图）

**验收**：大屏三张图（饼/条/折线）渲染正常、有 tooltip、空态优雅；指标随 SSE 实时变化；移动端宽度自适应。

## 三、依赖关系与实施顺序

```
任务2(ES事件仓) ──┬──► 任务4(consumer接主图，事件可见/持久)
                 ├──► 任务5(SSE 事件总线，发布点依赖 store 变更)
                 └──► 任务6(OpenAPI schema 稳定后再生成前端类型)

任务1(Redis审批) ─┬──► 任务5(审批变更发布)
                 └──► 任务3(真实动作需审批闸门，已具备；超时兜底依赖 Celery)

任务3(真实动作)     独立，但审计/幂等依赖任务2
任务6/7(前端)       依赖后端 response_model/新端点稳定（任务2/5/7的 timeline 端点）
```

**建议实施顺序**：
1. **任务 2**（事件落 ES）→ **任务 1**（审批落 Redis）：多 worker / SSE 的地基，先做。
2. **任务 4**（consumer 接主图）：抽取 runner 后，Kafka 入口立刻可用。
3. **任务 3**（真实动作）：接 3 个可跑连接器 + dry-run 默认。
4. **任务 5**（SSE）：在 1/2 的发布点之上搭建，前端去轮询。
5. **任务 6 + 7**（前端收尾）：强类型/错误边界/E2E/图表，最后统一收敛。

**首步建议**：任务 2——落地 `ESEventStore` + 全节点审计落 ES + `/trace` 改查 ES，配测试并在 `--workers 3` 下验证。完成后多 worker 状态共享、SSE 事件源、Consumer 可见性三件事一并解锁。
