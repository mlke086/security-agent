# 主机漏洞扫描子系统开发方案

> 编制日期：2026-07-14
> 依据：`docs/企业级AI增强主机漏洞扫描系统设计方案.md` + 现有 security-agent 系统架构（`docs/项目梳理.md`）
> 目标：在现有系统上新增"控制台 → 主系统 → 子 Agent"的主机漏洞扫描能力，复用现有 FastAPI/React/LangGraph/Redis/ES/ModelAdapter/SSE/RBAC。

---

## 0. 目标与定位

在现有 security-agent（告警智能分析与响应）系统上，**新增一个主机漏洞扫描子系统**，形成"控制台发号施令、子 Agent 干活、全程可监控、AI 汇总出报告"的闭环。

**核心能力**：
- 主机纳管：目标主机执行一条 `curl` 命令 → 下载安装脚本 → 安装子 Agent → 自动注册并连回主系统。
- 扫描控制：控制台通过**对话式（自然语言）或手动表单**提交扫描任务 → 主系统向目标主机的 Agent 下发指令。
- 实时监控：Agent 扫描的**每一个步骤**实时回传，控制台可见。
- AI 汇总：扫描结果回传 → 主系统聚合 → LLM 分析（误报过滤/风险分级/修复建议）→ 生成漏扫报告。

**典型业务流**：
```
控制台发起漏扫 → 主系统创建任务+下发指令 → 子Agent本地扫描(每步上报)
→ 结果回传 → 主系统汇总 → LLM分析 → 生成漏扫报告 → 控制台实时展示+导出
```

**与现有系统的关系**：本子系统是**独立新增模块**，不改动现有告警分析主链路（entry→orchestrator→investigate/vuln_check→respond）。现有 `vuln_hunter` 子图是"PoC 生成+沙箱验证"（主动验证可利用性），与"主机本地漏洞扫描"（被动检测）互补但分离；后续可让 vuln_hunter 消费主机漏扫发现做可利用性验证。

## 2. 关键技术决策（推荐方案 + 备选）

| 决策 | 推荐方案 | 理由 | 备选 |
|---|---|---|---|
| Agent 语言 | **Go** | 单静态二进制、跨平台交叉编译、无运行时依赖、低资源占用，业界主机 Agent 主流（osquery/fleet 风格） | Python（PyInstaller 打包；团队熟悉但体积大） |
| 控制通道 | **WebSocket**（Agent 主动连出，持久，JSON） | NAT 友好（Agent 拨出）、双向实时（下发指令 + 回传每步）、契合"每步可监控" | gRPC 双向流（更强类型但部署重）；HTTP 长轮询（实时性差） |
| 扫描引擎 | **内置采集 + 规则包比对** | 本地采集系统状态（补丁/包版本/配置/端口/弱口令）+ 加密规则包（CVE/基线）比对，符合设计文档"本地规则包扫描" | 嵌入 osquery（强但加依赖）；包装 lynis/openvas |
| LLM 分析 | **复用 `ModelAdapter`（DeepSeek）** | 已有统一 LLM 适配，私有化可切 vLLM | 独立部署本地模型 |
| 实时推送 | **复用 SSE + Redis pub/sub** | 前端已有 SSE 通道，任务进度走 `vulnscan:{task_id}` 通道 | WebSocket（前端再开一路） |
| 主机纳管 | **curl 安装脚本 + 一次性纳管令牌** | 符合你的设想，零依赖、跨平台（bash/PowerShell） | Ansible/批量推送（需额外基础设施） |

> 这些是方案推荐，落地前可确认或调整；下文按推荐方案展开。

## 3. 主机纳管设计（你的纳管思路落地）

### 3.1 纳管令牌

- 管理员在控制台"主机纳管"页生成**纳管令牌（enrollment token）**：
  - 可绑定主机组、有效期、使用次数（默认一次性，可放开批量）。
  - 存 Redis（`enroll:{token}` -> {group, expires, uses}），用后即焚或倒计次数。

### 3.2 纳管命令（用户在目标主机执行）

- **Linux**：
  ```bash
  curl -fsSL https://console-host:8000/api/v1/agents/install?token=<TOKEN> | bash
  ```
- **Windows（PowerShell）**：
  ```powershell
  irm https://console-host:8000/api/v1/agents/install?token=<TOKEN> | iex
  ```

### 3.3 安装脚本逻辑（`GET /api/v1/agents/install`）

端点根据 `User-Agent`/参数返回 bash 或 PowerShell 脚本，脚本完成：
1. 探测 OS 与架构（linux/amd64、linux/arm64、windows/amd64）。
2. 从主系统下载对应 Agent 二进制（`GET /api/v1/agents/binary/{os}/{arch}`）。
3. 安装为系统服务（Linux: systemd unit；Windows: 服务注册）。
4. 写入配置（console URL、enrollment token）。
5. 调用 `POST /api/v1/agents/enroll` 注册 → 拿到 `agent_id` + 长期 `agent_token`。
6. 启动服务 → Agent 建立 WebSocket 连接。

### 3.4 注册接口（`POST /api/v1/agents/enroll`）

```
请求: { token, hostname, os, arch, ip, kernel_version }
处理: 校验 token -> 分配 agent_id + agent_token -> 落库(ES hosts) -> 返回
响应: { agent_id, agent_token, ws_url, heartbeat_interval }
```
- `agent_token` 长期凭证，用于 WS 连接鉴权（后续可轮换）。
- 注册成功后主机在控制台"主机列表"可见（初始在线）。

### 3.5 纳管后状态机

```
未纳管 --curl--> 安装中 --enroll--> 已注册 --WS连上--> 在线
   在线 --心跳超时--> 离线 --重连--> 在线
   在线/离线 --卸载--> 已下线
```

## 4. 通信协议设计

### 4.1 通道划分

| 通道 | 协议 | 用途 | 方向 |
|---|---|---|---|
| 纳管/文件 | REST | install 脚本、二进制下载、规则包下载 | 主系统 ↔ Agent |
| 控制/监控 | **WebSocket** | 指令下发、心跳、步骤上报、结果上报 | Agent 主动连出，双向 |
| 前端实时 | SSE（复用） | 任务进度推控制台 | 主系统 → 前端 |

### 4.2 WebSocket 消息协议（JSON）

连接：`wss://console-host:8000/api/v1/agents/ws?agent_id=...&token=...`

**主系统 → Agent（指令）**：
| type | 说明 | payload |
|---|---|---|
| `scan_command` | 下发扫描任务 | `{task_id, policy, rule_version, modules[], resource_limit, deadline}` |
| `scan_cancel` | 取消任务 | `{task_id}` |
| `rule_update` | 规则热更新 | `{rule_version, download_url, signature}` |
| `agent_upgrade` | 版本升级 | `{version, download_url, signature}` |
| `config_update` | 策略/配置更新 | `{heartbeat_interval, resource_limit}` |

**Agent → 主系统（上报）**：
| type | 说明 | payload |
|---|---|---|
| `heartbeat` | 心跳 | `{agent_id, ts, cpu, mem, rule_version, agent_version}` |
| `scan_step` | 扫描步骤进度 | `{task_id, step, status(running/done/failed), message, ts}` |
| `scan_result` | 扫描结果（分批/完成） | `{task_id, batch, findings[], is_final, ts}` |
| `task_ack` | 任务接收确认 | `{task_id, accepted}` |
| `update_ack` | 更新结果 | `{kind, version, ok, error}` |

### 4.3 安全

- **传输**：TLS（wss/https）；内网可自签 CA，Agent 内置 CA 证书。
- **身份**：enrollment token（纳管）→ agent_token（长期，WS 鉴权）→ 可升级为 mTLS。
- **指令签名**：`scan_command`/`rule_update`/`agent_upgrade` 携带主系统签名，Agent 校验后才执行（防伪造指令）。
- **结果加密**：扫描结果上报前 Agent 端加密/脱敏（不上传原始配置文件）。
- **断线续传**：WS 断开时 Agent 本地缓存任务与结果，重连后续传。

### 4.4 主系统侧连接管理

- `src/agents/ws_gateway.py`：维护 `agent_id -> WebSocket` 连接表（内存 + Redis 副本，支持多 worker）。
- 下发指令：按 `agent_id` 找连接 → `send_json`；连接不在本 worker → Redis pub/sub `agent:cmd:{agent_id}` 跨 worker 转发。
- Agent 上报：`scan_step`/`scan_result` → 落 ES + 发 Redis pub/sub `vulnscan:{task_id}` → SSE 推前端。

## 5. 子 Agent 开发（Go，`agent/` 目录）

### 5.1 模块结构

```
agent/
├── cmd/agent/main.go          # 入口
├── internal/
│   ├── comm/                  # WS 通信 + 心跳 + 断线重连
│   ├── enroll/                # 纳管注册（解析配置、调 enroll API）
│   ├── scan/                  # 本地扫描引擎
│   │   ├── collector.go       # 采集系统状态(包/补丁/端口/配置/账号)
│   │   ├── rules.go           # 规则加载与比对
│   │   ├── baseline.go        # 基线检查
│   │   └── weakpass.go        # 弱口令检测
│   ├── resource/              # CPU/内存/IO 限流
│   ├── report/                # 结果整理 + 加密上报
│   ├── updater/               # 规则热加载 + 自升级守护
│   └── config/                # 配置(console url/token/agent_id)
├── rules/                     # 规则包(加密, 下发)
└── packaging/                 # systemd unit / windows service / 安装脚本
```

### 5.2 核心行为

1. **启动**：读配置 → 若无 `agent_id` 则调 `enroll` 注册 → 拿凭证 → 建 WS 连接 → 进心跳循环。
2. **收到 `scan_command`**：`task_ack` → 限流起扫描 → 每个模块/步骤发 `scan_step` → 产出 findings → 分批发 `scan_result`（`is_final=true` 收尾）。
3. **扫描引擎**（本地完成，不发包）：
   - 系统漏洞：采集已装包/补丁版本 → 比对 CVE 规则包。
   - 中间件漏洞：探测 Nginx/Tomcat/Apache/Redis/MySQL 等版本 → 比对版本规则。
   - 安全基线：账号权限/端口/密码策略/文件权限/日志配置核查。
   - 弱口令：本地账号、数据库、服务的弱口令合规校验（仅本地校验，不爆破远程）。
4. **资源限流**：扫描前读 `resource_limit`（CPU%/Mem%/时间窗），超阈值自动降速/暂停，业务高峰避让。
5. **规则热更新**：收到 `rule_update` → 下载 → 校验签名 → 内存热加载（不重启）。
6. **自升级**：收到 `agent_upgrade` → 守护进程下载新版 → 校验 → 替换 → 重启。

### 5.3 跨平台构建

- `make build` 交叉编译：linux/amd64、linux/arm64、windows/amd64。
- 产物单二进制（无依赖），systemd/服务化安装由安装脚本完成。

### 6.2 新增 REST API

| 方法 | 路径 | 说明 | 鉴权 |
|---|---|---|---|
| GET | `/api/v1/agents/install` | 返回安装脚本（bash/ps1） | enrollment token |
| GET | `/api/v1/agents/binary/{os}/{arch}` | 下载 Agent 二进制 | token |
| POST | `/api/v1/agents/enroll` | 注册，返回 agent_id/token | enrollment token |
| GET | `/api/v1/agents` | 主机/Agent 列表（状态/版本） | admin/analyst |
| POST | `/api/v1/agents/enroll-tokens` | 生成纳管令牌 | admin |
| GET | `/api/v1/agents/{id}` | 主机详情 | admin/analyst |
| WS | `/api/v1/agents/ws` | Agent 通信网关 | agent_token |
| POST | `/api/v1/vulnscan/tasks` | 创建扫描任务（手动/对话解析后） | admin/analyst |
| POST | `/api/v1/vulnscan/tasks/parse` | 自然语言 → 任务参数（LLM） | admin/analyst |
| GET | `/api/v1/vulnscan/tasks` | 任务列表 | admin/analyst |
| GET | `/api/v1/vulnscan/tasks/{id}` | 任务详情+进度 | admin/analyst |
| GET | `/api/v1/vulnscan/tasks/{id}/stream` | SSE 实时进度 | admin/analyst |
| GET | `/api/v1/vulnscan/results` | 漏洞清单（聚合） | admin/analyst/viewer |
| GET | `/api/v1/vulnscan/reports/{task_id}` | 漏扫报告 | admin/analyst |

### 6.3 扫描任务调度（vulnscan 子图）

新增 LangGraph 子图 `vulnscan_graph`，节点：
```
parse_intent → dispatch → collect → aggregate → llm_analysis → report → END
```
- `parse_intent`（仅对话式）：LLM 把自然语言解析为 `{host_ids/groups, policy, modules, schedule}`。
- `dispatch`：创建 `ScanTask`（ES），向目标 Agent 逐个发 `scan_command`（WS），记待回收数。
- `collect`：异步收 `scan_result`（按 task_id 汇聚），收齐或超时 → 进入聚合。
- `aggregate`：合并多主机 findings，去重、标准化（CVE/等级/影响）。
- `llm_analysis`：复用 `ModelAdapter`，输入聚合 findings + 主机上下文 → 误报过滤 + 风险分级（CVSS×资产等级）+ 修复建议（补丁/缓解/回滚）。
- `report`：生成结构化漏扫报告（落 ES + 可导出），发 `vulnscan:{task_id}` 完成事件 → SSE。

### 6.4 数据模型（ES 索引）

| 索引 | 关键字段 |
|---|---|
| `vulnscan-hosts` | agent_id, hostname, ip, os, arch, status(online/offline), agent_version, rule_version, last_heartbeat, group, owner |
| `vulnscan-tasks` | task_id, source(dialog/manual), targets[], policy, status, created_at, finished_at, stats{total/done/failed} |
| `vulnscan-results` | task_id, agent_id, hostname, findings[], ts |
| `vulnscan-vulns` | vuln_id, task_id, host, cve, name, severity, ai_severity, status, fix_advice, owner |
| `vulnscan-reports` | task_id, summary, ai_analysis, stats, generated_at |

Redis：
- `agent:online:{agent_id}` 心跳 TTL（在线状态）。
- `agent:conn:{agent_id}` 连接所在 worker（多 worker 路由）。
- `agent:cmd:{agent_id}` pub/sub 跨 worker 下发。
- `vulnscan:{task_id}` pub/sub → SSE。
- `enroll:{token}` 纳管令牌。

### 6.5 实时监控链路

```
Agent scan_step/scan_result --WS--> ws_gateway --写ES + 发Redis pub/sub vulnscan:{task_id}
                                                          │
                          前端 SSE /api/v1/vulnscan/tasks/{id}/stream 订阅 ←┘
                          → 实时刷新每台主机步骤/进度/结果
```

## 7. 前端控制台集成

### 7.1 新增页面（`frontend/src/pages/`）

| 页面 | 路由 | 功能 |
|---|---|---|
| 主机纳管 | `/hosts` | 生成纳管令牌、显示安装命令（一键复制 curl）、主机列表（在线/版本/规则版本/最后心跳）、离线告警 |
| 扫描任务 | `/vulnscan/tasks` | **对话式**输入框（NL → 预览解析参数 → 确认提交）+ **手动表单**（选主机/组、策略、模块、时间窗、限流）+ 任务列表 |
| 扫描监控 | `/vulnscan/monitor/:taskId` | 实时进度：每台主机步骤时间线（running/done/failed）、完成数/失败数、findings 实时流入 |
| 漏洞清单 | `/vulnscan/vulns` | 聚合漏洞列表（CVE/等级/AI 等级/状态/责任人），筛选、导出 |
| 漏扫报告 | `/vulnscan/reports/:taskId` | AI 分析摘要 + 风险分布 + 修复建议 + 导出（PDF/Word/Excel） |

### 7.2 对话式任务提交

- 页面输入："扫描生产环境所有 Linux 主机的系统漏洞和基线，限流 30% CPU"。
- `POST /api/v1/vulnscan/tasks/parse` → LLM 解析为 `{groups:[生产], os:linux, modules:[sys_vuln,baseline], resource_limit:{cpu:30}}`。
- 前端展示解析结果（可编辑）→ 确认 → `POST /api/v1/vulnscan/tasks`。
- 复用现有 antd + ErrorBoundary + openapi 强类型（新端点加 `response_model` → `gen:types` 自动生成）。

### 7.3 实时监控 UI

- 复用现有 SSE 模式（EventSource 订阅 `/vulnscan/tasks/{id}/stream`）。
- 每台主机一行：状态灯 + 步骤 Timeline（采集→比对→基线→弱口令→上报）+ findings 计数。
- 任务完成自动跳报告页或弹出查看。

### 7.4 导航与权限

- `AppLayout` 菜单新增：主机纳管、扫描任务、漏洞清单（admin/analyst 可见；viewer 只读漏洞清单与报告）。
- 复用 RBAC `require_role`。

## 8. 端到端业务流程

### 8.1 主机纳管时序

```
管理员(控制台) --生成令牌--> 主系统(Redis enroll:{token})
管理员在目标主机: curl .../install?token | bash
  └─ 脚本: 探测OS → 下载二进制 → 装服务 → POST /enroll(token+主机信息)
        └─ 主系统: 校验token → 分配 agent_id+agent_token → 落ES(hosts) → 返回凭证
  └─ Agent 启动 → WS 连接(agent_token鉴权) → 心跳 → 控制台主机列表显示"在线"
```

### 8.2 漏洞扫描全流程（核心）

```
1. 控制台发起: 对话式(NL→LLM解析) 或 手动表单 → POST /vulnscan/tasks
2. 主系统创建 ScanTask(ES) → vulnscan_graph 启动:
3. dispatch: 向目标 Agent 逐个 WS 下发 scan_command(策略+规则版本+限流)
4. Agent: task_ack → 本地限流扫描:
     - 每步发 scan_step(采集/比对/基线/弱口令/上报) → 主系统实时转发前端
     - 产出 findings → 分批 scan_result(is_final 收尾)
5. collect: 主系统按 task_id 收齐(或超时)
6. aggregate: 合并多主机 findings, 去重/标准化
7. llm_analysis(ModelAdapter): 误报过滤 + 风险分级(CVSS×资产等级) + 修复建议
8. report: 生成漏扫报告(ES) → SSE 推完成 → 控制台展示 + 导出
9. (闭环) 漏洞清单 → 工单/整改 → 复测任务(复用策略) → 归档
```

### 8.3 规则更新与 Agent 升级流程（设计文档要求）

- 规则热更新：控制台同步规则库 → 灰度下发 `rule_update` → Agent 下载+校验签名+内存热加载 → `update_ack`。
- Agent 升级：控制台上传新版包 → 配置灰度/定时 → `agent_upgrade` → Agent 守护进程下载+替换+重启 → 结果回传。

## 11. 安全与运维考量

- **业务零冲击**：Agent 本地扫描 + 资源限流 + 时间窗避让，无主动探测发包。
- **无脚本下发**：常规扫描走本地规则包；应急脚本走白名单+签名+超时+审计（设计文档 4.2.6）。
- **数据最小化**：仅上报 findings，不上传主机原始配置/文件；结果加密。
- **可审计**：纳管、令牌、任务、下发、更新、结果全程落 ES audit，满足 6 个月留存。
- **多 worker**：WS 连接表 + Redis pub/sub 跨 worker 路由指令（与现有审批队列多 worker 方案一致）。
- **离线/断网**：Agent 缓存任务与结果，重连续传；离线主机自动标记+告警。

## 12. 风险与待确认

1. **Agent 语言**：推荐 Go（单二进制）。若团队仅 Python，改 Python+PyInstaller，体积/资源略增。**需确认**。
2. **扫描引擎深度**：MVP 先做"系统漏洞（包/补丁比对 CVE）+ 安全基线"；中间件/弱口令/数据库规则后续补。规则库来源（CVE/CNNVD 同步 vs 离线包）**需确认**。
3. **通信安全**：内网可先用 token+TLS；生产建议 mTLS + 指令签名。
4. **规则包签名**：需一套轻量签名/校验机制（主系统私钥签、Agent 公钥验）。
5. **二进制分发**：主系统托管 Agent 二进制（`/agents/binary/...`），或走对象存储（MinIO，服务器已有）。

---

## 13. 总结

本方案以"控制台 + 主系统 + 子 Agent"三层架构把主机漏洞扫描接入现有 security-agent 系统：**curl 一键纳管主机**、**WebSocket 实时控制与监控**、**LangGraph 子图调度 + ModelAdapter 做 LLM 分析**、**复用现有 ES/Redis/SSE/RBAC/前端框架**，形成"发起扫描 → Agent 本地扫描 → 结果回传 → 汇总 → AI 分析 → 报告"的完整闭环，且不扰动现有告警分析主链路。开发按 S1→S5 分阶段交付，每阶段可独立验证。
