## 二、仍影响功能的问题

### P1 - 严重

#### P1-1 DashboardPage @ant-design/charts 渲染崩溃风险
**文件**: `frontend/src/pages/DashboardPage.tsx:4,79,91,105`
**问题**: DashboardPage 仍用 `@ant-design/charts` 的 Pie/Column/Line。ScanReportPage 之前因该库动态加载崩溃已改成 antd Progress。DashboardPage 在数据为空（verdictData/priorityData/timeline 空）或库动态加载失败时同样会白屏崩溃。
**影响**: 首页运营大屏可能白屏，用户进系统第一眼就崩。
**修复**: 改用 antd Progress/Table 展示（同 ScanReportPage 的处理方式），或加 ErrorBoundary 包裹图表区域。

#### P1-2 EventQueuePage/EventDetailPage SSE cleanup 竞态
**文件**: `frontend/src/pages/EventQueuePage.tsx:47-75`, `frontend/src/pages/EventDetailPage.tsx:44-66`
**问题**: 这两个页面的 SSE 创建是 async（getSseToken 后），cleanup 用闭包变量 `es`。虽然加了 `if (cancelled) return` 和创建后检查，但 React 严格模式或快速切换时仍有极小竞态窗口（getSseToken 完成到 new EventSource 之间）。ScanMonitorPage 已用 eventSourceRef 修复，这两个页面没用 ref。
**影响**: 极端情况下 EventSource 泄漏（概率低，但事件队列页频繁进出可能触发）。
**修复**: 统一用 useRef 模式（同 ScanMonitorPage）。

### P2 - 一般

#### P2-1 vite proxy SSE 缓冲（dev 模式）
**文件**: `frontend/vite.config.ts`
**问题**: vite proxy `/api` 无 SSE 特殊配置，dev 模式下 SSE 事件可能被 proxy 缓冲，导致实时事件略有延迟。
**影响**: dev 模式实时性略差，不影响功能正确性。生产 nginx 需配 `proxy_buffering off`。
**修复**: vite.config.ts 加 `configure: (proxy) => proxy.on('proxyRes', (proxyRes) => proxyRes.headers['x-accel-buffering'] = 'no')`，或生产 nginx 配 `proxy_buffering off`。

#### P2-2 ScanMonitorPage 已完成任务重复拉取 findings
**文件**: `frontend/src/pages/ScanMonitorPage.tsx:24-30`
**问题**: refreshTask 在 completed/failed 时拉 findings。但 SSE onmessage 的防抖刷新也会调 refreshTask，可能重复拉 findings。
**影响**: 多一次 ES 查询，不影响功能。
**修复**: findings 拉取加 state 标记，已拉过不重复拉。

#### P2-3 agent rule_version 心跳上报与 PG 不同步
**文件**: `src/agents/manager.py:46-48`
**问题**: 心跳上报 rule_version 写 PG，但 agent 重启后从 config.json 读 rule_version。如果 config.json 的 rule_version 与 PG 不一致（如手动改过），心跳上报的版本与 PG 记录冲突。
**影响**: 可能触发不必要的规则下发。
**修复**: 低优先级，正常流程不会不一致。

#### P2-4 对话式扫描 intent 解析依赖 LLM 准确性
**文件**: `src/api/routers/scan_chat.py:280-286`
**问题**: 对话式扫描的 intent 解析完全依赖 LLM（DeepSeek）。LLM 解析不准时，targets/modules 可能错误。
**影响**: 对话式扫描意图解析可能不准，需用户在 IntentCard 确认。
**修复**: IntentCard 已允许用户确认/修改，可接受。

#### P2-5 主机组删除后 hosts.group_name 残留
**文件**: `src/agents/store.py:270-279`
**问题**: delete_group 删 host_groups 表行，但 hosts.group_name 不清空（保留组名）。list_groups 的 UNION 查询会把这些当 legacy 组显示。
**影响**: 删组后主机仍显示旧组名，组列表可能出现 legacy 孤儿。
**修复**: delete_group 时可选清空 hosts.group_name（或提示用户先迁移）。

### P3 - 建议

#### P3-1 main.py 预存 ruff E402/I001
**文件**: `src/api/main.py:23-25`
**问题**: `configure_logging()` 后再 import，ruff 报 E402/I001。预存问题，非本次引入。
**修复**: 重构 import 顺序（低优先级）。

#### P3-2 task_queue/* mypy strict 错误
**文件**: `src/orchestration/task_queue/*.py`
**问题**: 预存的 mypy strict 模式类型错误（redis-py/LangGraph 类型 stub）。非功能问题。
**修复**: 加 type: ignore 或修类型（低优先级）。

#### P3-3 GET 已删任务返回 200（Redis 缓存）
**文件**: `src/api/routers/vulnscan.py:124-156`
**问题**: 删除任务后 GET 仍可能返回 200（Redis `vulnscan:queue:status:{id}` 缓存回退）。
**影响**: 不影响任务列表（列表从 ES 查），仅直接 GET 已删任务时可能返回缓存。
**修复**: delete_task 时清 Redis 缓存。
