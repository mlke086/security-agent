## 一、项目概况

### 1.1 定位

基于 **LangGraph** 的企业级安全运营平台，融合两大能力：
1. **告警智能分析与自动响应**：告警接入 -> 研判/漏洞验证/HITL 审批 -> 处置执行，全程可审计、实时推送。
2. **主机漏洞扫描**：控制台 -> 主系统 -> 子 Agent 分布式架构，主机纳管 -> 本地扫描 -> AI 分析 -> 报告闭环。

### 1.2 当前状态

- **可运行、可验证、CI 全绿**：服务可启动，5/6 中间件可达，主链路端到端跑通。
- **质量基线**：ruff ✅ 0 错、mypy ✅ 0 错（strict）、后端单测 258 passed（memory 默认 + pg 模式均全过）、集成 10 passed + 4 skip、Go 8 包测试过、前端 tsc 干净 + Playwright 4/4。
- **持久化升级（Phase 1+2）**：核心数据已迁 PostgreSQL（PostgreSQL 17.6）--用户/RBAC、Agent 注册与认证令牌、HITL 审批记录、Agent 主机清单、事件记录均落 PG；Redis 降级为缓存/在线状态/pub-sub，ES 降级为全文检索镜像。
- **代码规模**：后端 104 个 Python 文件、Go Agent 10 个包、前端 11 个页面、29 个测试文件。

## 二、系统架构

### 2.1 总体架构

系统由**两大子系统**构成，共享 FastAPI 后端、React 前端、Redis/ES/Kafka/Milvus/Neo4j 中间件、RBAC、SSE、ModelAdapter：

```
┌──────────────────────── 控制台(前端 React, 11 页面) ────────────────────────┐
│ 告警: Login/Dashboard/EventQueue/EventDetail/Approvals                       │
│ 漏扫: HostOnboard/ScanTask/ScanMonitor/VulnList/ScanReport/Rules             │
└───────┬──────────────────────────────────────────────────┬─────────────────┘
        │ REST + SSE                                        │ REST + SSE
┌───────▼──────────────────────────────────────────────────▼─────────────────┐
│                    主系统(FastAPI 后端, src/)                                 │
│ ┌──────────── 子系统A：告警分析 ────────────┐  ┌──── 子系统B：漏洞扫描 ────┐ │
│ │ preprocessing(脱敏/IOC) -> main_graph     │  │ agents(纳管/通信/规则)    │ │
│ │   -> investigate/vuln_hunter/responder   │  │ vulnscan 子图(调度/AI)    │ │
│ │ execution(沙箱/动作)                      │  │                           │ │
│ └──────────────────────────────────────────┘  └───────────────────────────┘ │
│ 共享: api(routers) · common(db(PG)/audit/config/logging/signing) · knowledge(GraphRAG/tools) │
│       ModelAdapter(LLM) · EventBus/SSE · RBAC · PostgreSQL(主存) · Redis(缓存/pub-sub) · ES(检索镜像) · Milvus · Neo4j │
└───────────────────────────┬────────────────────────────────────────────────┘
                            │ WebSocket(agent 主动连出, TLS, Ed25519 验签)
        ┌───────────────────┼───────────────────┐
   ┌────▼────┐          ┌────▼────┐         ┌────▼────┐
   │ Go Agent│          │ Go Agent│   ...   │ Go Agent│  (每台主机一个)
   │ 本地扫描│          │ 本地扫描│         │ 本地扫描│
   └─────────┘          └─────────┘         └─────────┘
```

## 四、业务流程

### 4.1 告警智能分析链（子系统A）

```
告警(Kafka raw-alerts / HTTP /api/v1/events / demo seed)
  -> 脱敏(sanitize) + IOC 提取(IP/域名/hash/URL, 排除私有IP)
  -> entry(校验+审计) -> orchestrator(蜜罐规则优先, 否则 LLM 分流 priority/tags)
  -> route_decision:
       low -> ignore(归档)
       vulnerability tag -> vuln_check
       high/medium -> investigate
  -> investigate: cti_analyst(VT+GraphRAG+LLM IntelCard) -> investigator(LLM verdict+conf)
  -> aggregator -> route_after_verdict:
       stage=verify(来自vuln_check): true_positive->respond / 否则->done
       true_positive & conf>=0.8 -> respond
       0.5<=conf<0.8 -> vuln_check
       <0.5 -> done(归档)
  -> vuln_check: poc_generator(LLM) -> linter -> sandbox(Docker) -> 收敛(<=10轮)
  -> respond: playbook_matcher(规则/LLM) -> hitl_approval:
       L1/L2 -> 自动批准
       L3+ -> Redis 审批(quorum: L3=1, L4/L5=2) + webhook 通知 + wait_result(pub/sub)
         approved -> execute_response(dispatcher: dry_run/幂等/回滚, 连接器)
         rejected/timeout -> reject_response
全程: 事件/轨迹->ES, 审计->ES, 审批->Redis, 状态变更->Redis pub/sub->SSE->前端实时
```

典型路径：蜜罐 `whoami` -> high -> investigate -> benign -> 归档；高危 CVE -> vuln_check -> TP -> respond -> 剧本 -> 审批 -> 执行。

### 4.2 主机漏洞扫描链（子系统B）

```
[纳管] 控制台生成纳管令牌 -> 目标主机 curl .../install?token | bash
  -> 脚本探测OS/下载二进制/装服务/调 enroll -> Agent 拿凭证建 WS(TLS+token) -> 心跳上线 -> 控制台可见

[扫描] 控制台发起(对话式 NL->LLM 解析 / 手动表单) -> POST /vulnscan/tasks
  -> 主系统建 ScanTask(ES) -> vulnscan 子图:
       dispatch: 向目标 Agent WS 下发 scan_command(策略+规则版本+限流, Ed25519 签名)
       Agent: task_ack -> 本地限流扫描:
         collector(包/补丁/账号/端口/密码/文件) -> matcher(dpkg 版本比较 + config_check)
         每步发 scan_step -> 产出 findings -> 分批 scan_result(is_final)
       collect: 主系统按 task_id 收齐(或超时)
       aggregate: 合并多主机 findings, 去重标准化
       llm_analysis(ModelAdapter): 误报过滤 + 风险分级(CVSS×资产) + 修复建议
       report: 生成漏扫报告(ES) -> SSE 推完成 -> 控制台展示+导出
  -> (闭环) 漏洞清单 -> 工单/整改 -> 复测 -> 归档
```

### 4.3 实时监控链路

Agent `scan_step`/`scan_result` --WS--> ws_gateway --写 ES + 发 Redis pub/sub `vulnscan:task:{id}` --> 前端 SSE `/vulnscan/tasks/{id}/stream` 订阅 -> 实时刷新每主机步骤/进度/结果。告警侧同理（`events:{id}`/`metrics` 通道）。

## 五、数据与存储

### 5.1 PostgreSQL（主存储，Phase 1+2 迁移，7 张表）

| 表 | 用途 | 原存储 |
|---|---|---|
| `users` | 用户 & RBAC（username/hashed_password/role/disabled），启动种子 admin/analyst/viewer/responder | 内存 dict（重启全丢） |
| `enroll_tokens` | Agent 注册令牌（token/group/expires/uses/created_by/used_at，可审计） | Redis Hash（无审计） |
| `agent_tokens` | Agent 认证令牌（agent_id/token_hash(SHA-256)/issued_at/revoked_at，可吊销） | Redis setex（无审计） |
| `approvals` | HITL 审批（approval_id/event_id/status/required/operation_level/created_at/resolved_at） | Redis Hash+Set+ZSet |
| `approval_votes` | 审批投票审计（approval_id/voter/decision，UNIQUE 防重复） | Redis Set（无决策记录） |
| `hosts` | Agent 主机清单（agent_id/hostname/ip/os/status/group/version/heartbeat，索引 status/group） | ES（无事务） |
| `events` | 告警事件（event_id + JSONB data 存整个 EventRecord，索引 status/submitted_at） | 内存 dict / ES |

**设计点**：
- 事件用 **JSONB** 存整个 EventRecord，支持 `data->>'status'` 索引查询、`jsonb_set` 原子追加 trace/approvals。
- Host **双写**：PG 同步事务写主存储，ES 异步 fire-and-forget 镜像（PG 挂 fallback ES 读）。
- Agent 令牌存 **SHA-256 哈希**（非明文），支持 `revoked_at` 吊销。
- 审批 **PG 事务原子 quorum**（INSERT vote + UPDATE status），`approval_votes` UNIQUE 约束防重复投票。
- PG 连接池**事件循环感知**（跨循环自动重建，解决 TestClient/pytest-asyncio 多循环冲突）。
- 无新依赖（仅 Phase 1 的 asyncpg，无 ORM）。

### 5.2 Redis（缓存/在线状态/pub-sub，降级使用）

| Key | 类型 | 用途 |
|---|---|---|
| `approval:notify:{id}` | pub/sub | 审批实时通知（wait_result） |
| `agent:online:{id}` | String TTL | Agent 在线状态（心跳） |
| `agent:conn:{id}` | String | WS 连接所在 worker（跨 worker 路由） |
| `agent:cmd:{id}` | pub/sub | 跨 worker 下发指令 |
| `events:{id}`/`events:list`/`metrics` | pub/sub | 事件总线 -> SSE |
| `action:done:{op_id}` | String | 动作幂等 |
| `rules:current_version` | String | 规则版本缓存 |

### 5.3 Elasticsearch（检索/审计镜像）

| 索引 | 用途 |
|---|---|
| `security-agent-events` | 事件全文检索（store_backend=es 时） |
| `security-agent-audit` | 审计日志（每节点一步一 doc，`/trace` 聚合） |
| `vulnscan-hosts` | Host 搜索镜像（PG 主存的镜像） |
| `vulnscan-tasks/results/vulns/reports` | 漏扫任务/结果/漏洞/报告（仍 ES 主存） |

### 5.5 store_backend 切换

`settings.store_backend`（默认 `es`）：
- `pg` -- PostgreSQL JSONB 主存（Phase 2，事件持久化）
- `memory` -- 内存（离线/测试，开箱即用）
- `es` -- Elasticsearch（全文检索）
测试默认 `memory`（`STORE_BACKEND=pg` 切 PG 路径，conftest 自动 TRUNCATE 隔离）。

## 七、遗留问题

### 7.1 功能/数据类（影响能力）

1. **Milvus 向量库近乎空（6 条）**--GraphRAG 向量召回实际无内容，研判情报向量路为空。需跑 `ingest_knowledge.py` 真实入库（ATT&CK/CVE/历史报告）。
2. **vuln_hunter 端到端未实测**--沙箱镜像已构建（服务器上），但本机无 Docker daemon，poc->linter->sandbox->收敛闭环未跑通验证（集成测试跳过 vuln_check）。
3. **MemoryManager.cleanup() 空转**--扫描从未写入的 `event:ts:*` Redis key，不清 Milvus/Neo4j，TTL 清理形同虚设（证据已迁 PG/Neo4j，但 cleanup 逻辑未跟进）。

### 7.2 部署/环境类

4. **沙箱镜像仅在服务器**--本机无 Docker，本机跑的 app 无法用沙箱；需 app 部署到服务器或本机装 Docker。
5. **embedding `local_files_only=True`**--需 BGE 模型本地缓存；新机器未预装时 `embed()` 失败，GraphRAG 静默降级。
6. **`.env` 只有 JWT+LLM**--中间件（含 PG）走 settings 默认值（可用），生产建议显式配置 + KMS 注入 `API_SECRET_KEY`/PG 密码。

### 7.3 测试覆盖类（非阻断）

7. **覆盖率未达 80%**--核心逻辑已 80%+，剩 main_graph/graph(25%)、runner(32%)、connectors(29-39%)、graphrag(28-55%)、celery(0%) 需补。
8. **Go `cmd/agent`、`version` 无测试**（价值低）。
9. **漏扫 E2E 6 场景待补**（纳管/扫描/监控/规则更新/离线重连/权限）。
10. **集成 4 skip**：沙箱未在本机构建、responder langgraph 0.2.28 无 interrupt、graphrag 需入库数据。

### 7.4 已知设计限制（非 bug）

11. **vLLM 未用**--走 DeepSeek 远程 API；私有化可经 ModelAdapter 切 vLLM。
12. **真实处置连接器有限**--notify/siem_tag/dns_block 真实；firewall/isolate/edr 为 simulator 占位。
13. **responder 用轮询式 HITL**（非 LangGraph interrupt/resume）--MVP 方案，生产建议升级 checkpointer + interrupt。
14. **漏扫 tasks/results/vulns/reports 仍走 ES**--仅 Host 迁了 PG，其余漏扫数据 Phase 2 未迁（按需后续迁）。
15. **测试 teardown `Event loop is closed` 告警**--本地集成测试末尾 redis `__del__`，CI 不触发，彻底消除需 pytest loop-scope 改 session。

## 八、整体评估

### 8.1 成熟度

项目从"起不来的脚手架"已进入"**可运行、可验证、CI 全绿、核心链路打通、双子系统成型、持久化落地**"阶段：
- **告警分析链**：端到端跑通（蜜罐事件->研判->归档；高危->响应->审批->执行），多 worker 共享（PG+Redis），SSE 实时，HITL 闭环。
- **漏洞扫描链**：架构完整（控制台->主系统->Go Agent），纳管/通信/调度/AI 分析/报告全链路代码就绪，Go agent 能构建/测试（8 包）、dpkg 版本比较、Ed25519 验签、TLS/resource/queue 接入。
- **持久化（Phase 1+2）**：核心数据（用户/令牌/审批/主机/事件）已落 PostgreSQL，重启不丢、多 worker 共享、可审计；Redis 降为缓存/pub-sub，ES 降为检索镜像。无内存单点。

### 8.2 代码质量

- **静态全绿**：ruff 0、mypy 0（strict，放宽 4 注解标志 + adapter per-module）、tsc 0、go vet 0。
- **测试充分**：258 单测（memory + pg 双模式全过）+ 10 集成 + Go 8 包 + Playwright 4，核心逻辑覆盖 80%+。
- **类型一致**：openapi codegen（response_model -> schema.ts -> types.ts 单一真相源）。
- **安全**：JWT RBAC（PG users）、Ed25519 指令签名、Agent 令牌 SHA-256 哈希+可吊销、TLS、沙箱隔离、审计全程落 ES。
- **持久化健壮**：PG 连接池事件循环感知、Host 双写(PG 主+ES 镜像)带 fallback、事件 JSONB 索引查询。

### 8.3 主要风险/待办（按优先级）

1. **数据入库**（Milvus 向量库填充）--阻断研判向量召回评估。
2. **vuln_hunter 端到端验证**--需沙箱在本机或服务器跑 app。
3. **cleanup 实现**--MemoryManager TTL 清理（跟进 PG/Neo4j 迁移）。
4. **覆盖率提升**--补 graph/runner/connectors/celery，向 80% 推进。
5. **漏扫 E2E**--6 场景补齐，端到端验证。
6. **漏扫数据迁 PG**（可选）--tasks/results/vulns/reports 当前仍走 ES。
7. **生产加固**--沙箱网络严格隔离、Prompt 注入测试、压测、TLS 配置、PG 密码 KMS 注入。

### 8.4 结论

项目工程化程度较高，双子系统（告警分析 + 漏洞扫描）架构清晰、代码模块化、测试与 CI 门禁完善，**持久化已落地 PostgreSQL**（消除内存单点、重启不丢、多 worker 共享、可审计）。剩余问题集中在**数据入库、端到端验证、覆盖率提升**三项功能完整性/质量工作，以及沙箱部署的环境约束。代码层无阻断 bug，可进入联调与 E2E 验证阶段，向生产就绪推进。
