## 一、项目概况

基于 **LangGraph** 的安全事件智能分析与自动响应 Agent。告警接入后，经"研判 -> 漏洞验证 -> 响应处置"多子图协作，产出结论/置信度/PoC，并按风险等级走 HITL 审批后执行处置动作，全程可审计、实时推送。

- 中间件：`192.168.80.101`（主机名 Rocky001）Docker 部署，SSH `root/615700`
- 版本：0.1.0，当前处于"可运行、可验证、CI 全绿"阶段

## 四、详细功能（按模块）

### 1. 接入与预处理
- Kafka 消费：sanitize（脱敏密码/PII/hash）-> IOC 提取（IP/域名/hash/URL，排除私有 IP）-> `run_pipeline`，模块级 `Semaphore(4)` 限流；解析失败进 DLQ + commit，管线失败不 commit（重投）。
- HTTP 提交：`POST /api/v1/events?sync=`（sync 同步跑管线，否则 BackgroundTask）。
- 演示种子：`/api/v1/demo/seed` 注入 5 条样本事件（含 L3 待审批）。

### 2. 编排主图（`main_graph/graph.py`）
- `entry`：校验事件、写审计。
- `orchestrator`：蜜罐规则（whoami/id 等命令 -> high + honeypot）优先，否则 LLM 分流（priority/tags/noise）。
- `route_decision`：low -> ignore；`vulnerability` tag -> vuln_check；high/medium -> investigate。
- `aggregator`：汇聚子图结果，写审计。
- `route_after_verdict`：`stage=verify` -> (true_positive? respond : done)；首次研判：true_positive & conf≥0.8 -> respond，0.5–0.8 -> vuln_check，<0.5 -> done。
- 各子图带超时（investigate 120s / vuln 600s / respond 300s，可被 `hitl_timeout_sec` 覆盖）。

### 3. 子图
- **investigation**：`cti_analyst`（并行查 VT + GraphRAG 本地召回 + LLM 出 IntelCard，证据入 MemoryManager）-> `investigator`（CoT 提示，LLM 出 verdict/confidence/evidence/mitre_ttps）。
- **vuln_hunter**：`poc_generator`（LLM 生成 PoC）-> `poc_linter`（AST 危险调用/导入白名单/别名/getattr 拦截）-> `SandboxExecutor`（Docker 隔离执行，read_only/no-new-privileges/internal 网络）-> 收敛判断（≤10 轮）。
- **responder**：`playbook_matcher`（10 个 YAML 剧本规则匹配，`utf-8-sig` 读避免 BOM；未命中走 LLM 生成）-> `hitl_approval`（L1/L2 自动；L3+ Redis 审批，Lua 原子 quorum：L3=1 人、L4/L5=2 人；企微/钉钉推送审批卡；`wait_result` 循环到终态，pub/sub 通知）-> `execute_response`/`reject_response`。

### 4. 处置动作（`execution/actions`）
- `ActionDispatcher`：dry_run 默认 True；幂等（Redis `action:done:{op_id}`，**先查存在 -> 成功后 mark**，失败可重试）；逐 op 审计；失败回滚已成功步。
- 连接器：`notify`(企微/钉钉)、`siem_tag`(ES 打标)、`dns_block`(Redis blocklist) 真实可跑；`firewall_block/isolate_host/terminate_process/quarantine_file/edr_policy` 走 `SimulatorConnector`（dry_run 占位）。

### 5. 存储与实时
- 事件仓：`ESEventStore`（ES `security-agent-events`，`store_backend=es` 默认；测试用 memory）。`update_event`/`add_trace_step` 同时 publish 到 Redis 总线。
- 审计：`AuditLogger` -> ES `security-agent-audit`（各节点 `audit.log`，失败不阻断主链）。
- 审批：`ApprovalStore` -> Redis（Hash + Set + ZSet + Lua + pub/sub）。
- SSE：`stream.py` 三个端点（事件流/指标流/列表流），JWT via `?token=`，心跳 15s。
- 事件总线：`EventBus`（Redis pub/sub）跨 worker fan-out。

### 6. 前端
- 页面：Login / Dashboard（4 统计卡 + Pie/Column/Line 趋势图 + SSE 实时）/ EventQueue / EventDetail（推理链 Timeline + 审批 + SSE 实时）/ Approvals。
- 强类型：`schema.ts` 由 OpenAPI 生成（`npm run gen:types` + `scripts/gen_openapi.py`），`types.ts` 重导出为单一真相源。
- ErrorBoundary 包根；axios 401 拦截器（登录端点 401 不刷新，保留错误提示）；Playwright E2E 4 场景。

## 五、业务流程（端到端）

```
告警(Kafka/HTTP) -> 脱敏+IOC提取 -> entry -> orchestrator(分流)
   ├─ low/noise -> ignore(归档)
   ├─ vulnerability -> vuln_check(poC↔linter↔sandbox收敛) -> aggregator
   │       └─ verify: true_positive->respond / 否则->done
   └─ high/medium -> investigate(cti->investigator verdict+conf) -> aggregator
           └─ conf≥0.8(TP) -> respond
              0.5–0.8 -> vuln_check
              <0.5 -> done(归档)

respond: playbook_matcher(规则/LLM) -> hitl_approval
   ├─ L1/L2 -> 自动批准
   └─ L3+ -> Redis审批(quorum) + webhook通知 + wait_result(pub/sub)
        ├─ approved -> execute_response(dispatcher: dry_run/幂等/回滚, 连接器)
        └─ rejected/timeout -> reject_response

全程: 事件/轨迹->ES, 审计->ES, 审批->Redis, 状态变更->Redis pub/sub->SSE->前端实时
```

典型路径示例：
- 蜜罐 `whoami` -> orchestrator 规则命中 high -> investigate -> verdict=benign/conf=0.95 -> aggregator -> 非 TP -> done（归档）。
- 高危 CVE exploit -> vuln_check -> 验证 TP -> respond -> 命中 cve_exploit 剧本(L3) -> 审批 -> 执行 firewall_block(simulator)/notify。

## 九、残留问题

### A. 功能/数据类（影响能力）
1. **Milvus 向量库近乎空（6 条）**——GraphRAG 向量召回实际无内容，研判情报向量路为空，M2 召回率无法评估。需跑 `scripts/ingest_knowledge.py` 真实入库（ATT&CK/CVE/历史报告）。
2. **vuln_hunter 端到端未实测**——沙箱镜像已构建（服务器上），但本机无 Docker daemon，poc->linter->sandbox->收敛的完整闭环未跑通验证（集成测试跳过 vuln_check）。
3. **MemoryManager.cleanup() 仍是空转实现**——扫描从未写入的 `event:ts:*` Redis key，且不清理 Milvus/Neo4j，TTL 清理形同虚设（`后续计划V2` 修复7 未做）。

### B. 部署/环境类
4. **沙箱镜像仅在服务器**——本机无 Docker，本机跑的 app 无法用沙箱；需 app 部署到服务器（或本机装 Docker）才能跑 vuln_hunter。
5. **embedding `local_files_only=True`**——需 BGE 模型本地缓存；新机器未预装模型时 `embed()` 失败，GraphRAG 静默降级。部署需预下载模型。
6. **`.env` 只有 JWT+LLM**——中间件走 `settings.py` 默认值（与服务器一致，可用），但生产建议显式配置 + KMS 注入 `API_SECRET_KEY`。

### C. 质量/测试类（非阻断）
7. **覆盖率 52%（未达 80%）**——核心 agent 逻辑已测，剩 graphrag/vuln_hunter/memory/api 路由需重 mock（Milvus/Neo4j/Docker/LLM 迭代）；CI 不设覆盖率门禁。
8. **测试 teardown `Event loop is closed` 告警**——本地集成测试末尾 redis `__del__` 触发；CI 只跑单测不触发；彻底消除需把 pytest loop-scope 改 session（有破坏绿套件风险，未做）。
9. **mypy 放宽了 4 个注解强制标志**——`disallow_any_generics/untyped_defs/untyped_calls`、`warn_return_any` 关闭 + adapter per-module 放宽；新增未注解函数不再强制标注（bug 检测保留）。恢复严格需逐文件补标注。

### D. 已知设计限制（非 bug）
10. **vLLM 未用**——走 DeepSeek 远程 API；私有化可经 `ModelAdapter` 切 vLLM。
11. **真实处置连接器有限**——仅 notify/siem_tag/dns_block 真实可跑；firewall/isolate/edr 等为 simulator 占位（需对接 EDR/防火墙 API）。
12. **responder 子图用轮询式 HITL**（非 LangGraph interrupt/resume）——MVP 方案，生产建议升级为 checkpointer + interrupt。
